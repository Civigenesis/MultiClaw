# MultiClaw / Civigenesis 概念设计 2.0

> 在《概念设计》第一版与《执行方案》基础上，吸收 MultiClaw 一期方案中适用于企业级服务的核心概念（Agent 通信、可观测性、记忆与压缩、故障恢复、技能与资源等），形成完整的概念与模块级设计。**本版仅设计概念、模块与架构，不涉及具体代码与接口实现。**

---

## 一、文档目的与版本说明

- **目的**：为 MultiClaw / Civigenesis 提供统一的概念基线，明确「管理员实例 + 多实例 + 单实例多实体」与 ZeroClaw 现状的差距，并纳入企业级服务所需的核心能力（通信、可观测、记忆、故障恢复、技能、资源），供后续详细设计与执行方案迭代使用。
- **2.0 变更**：保留第一版的管理员实例、多实例、单实例多实体与 CEO 机制；**新增** Agent 通信、可观测性、记忆与上下文管理、故障恢复与健康、技能与扩展、资源与配额等模块的概念与架构描述；与《执行方案》中的 ZeroClaw 现状分析一致，不引入与实例管理、初始化、技能使用相悖的旧设计。
- **范围**：概念、模块边界、架构分层与数据流；不含协议细节、数据结构定义与代码级设计。

---

## 二、基于 ZeroClaw 的现状与目标

### 2.1 现状（与《执行方案》一致）

- **配置与实例**：单份 `Config`，单 `config.toml`，无 `instance_id`；Onboard 只生成一份配置；所有命令（Agent、Gateway、Daemon、Service、Channel）绑定到该单份配置。
- **运行形态**：单进程单 config；无「管理员实例」与「普通实例」之分；无单实例内多实体、无 CEO 抽象。
- **控制流**：`main` 解析 CLI → Onboard 或 `Config::load_or_init()` → 按子命令分发；无实例注册表、无引导层「先起管理员」的约定。

### 2.2 目标（与第一版概念设计一致）

- **全局层**：一个**管理员实例（Administrer）**，在默认初始化流程中创建并启动，作为用户数字分身与集群控制平面；仅用户可调用其实例管理能力。
- **实例层**：多个**普通实例（Instance）**，每实例独立配置与生命周期，由管理员创建/关闭/删除并配置约束；实例可向管理员汇报，不可调用管理员能力。
- **实体层**：单实例内**多智能体实体（Entity）**，由该实例的**CEO Agent** 在管理员授予的约束内设计目标与团队；平台不强制实体间拓扑，可提供预设组织模式。

下文在保持上述三层的基础上，补充企业级服务所需的概念与模块。

## 三、总体架构概览

系统在逻辑上分为**三层角色**与**多条横向能力线**：

- **三层角色**：全局层（管理员实例）→ 实例层（普通实例）→ 实体层（CEO + 实体）。
- **横向能力线**：Agent 通信、可观测性、记忆与上下文、故障恢复与健康、技能与扩展、资源与配额。各能力线在「用户—管理员—实例—CEO—实体」上的作用范围与数据流由各节分别说明。

角色关系与第一版一致：用户仅驱动管理员实例的实例管理能力；管理员拥有全局视图并接收实例汇报；实例由管理员管理，内部由 CEO 组织；CEO 在约束内自由设计团队与实体。

---

## 四、管理员实例（Administrer）

与《概念设计》第一版一致，仅作要点复述。

- **角色**：用户数字分身 + 集群全局管理员；在默认初始化流程中创建并启动。
- **能力**：查看全局实例基本信息、查看具体实例详情、创建/关闭/删除实例、配置实例约束（Agent 数量、调用量、资源限额等）；统称**实例管理能力**。
- **访问控制**：**仅用户**（通过受控入口）可调用实例管理能力；其他实例与 Agent 不得调用，只能向管理员**汇报**（进度、资源申请等）。
- **与可观测、资源、故障恢复的关系**：管理员是「全局可观测」与「全局资源/配额策略」的消费与执行入口；实例健康与恢复策略由管理员侧或平台层统一配置与触发（见第八、第十节）。

---

## 五、实例层（Instance）

与第一版一致，仅作要点复述。

- **定义**：对应一类实验或业务单元（如经济实验、文明实验）；每实例**单独配置**、独立生命周期；由管理员创建/关闭/删除并配置约束。
- **约束**：管理员可为实例配置 Agent 数量上限、调用量/配额、资源限额等；实例内 CEO 与实体须在约束内运作，超限可经汇报向管理员申请放宽。
- **与管理员**：实例不调用管理员能力；可向管理员汇报；管理员可查看实例配置、运行状态与内部运行记录。

---

## 六、单实例多实体与 CEO 机制

与第一版一致，并明确与「可观测」「记忆」「技能」的衔接。

- **实体**：实例内多个智能体实体，每实体可有独立配置（角色、目标、工具、模型等）；实体列表与关系由 CEO 或预设模板决定。
- **CEO Agent**：实例内唯一的**实例管理员**角色；负责全局目标设计、团队设计、资源分配；在管理员下发的约束内创建团队与实体；不拥有「实例管理能力」。
- **协作模式**：平台不固定实例内拓扑；可提供**预设模式**（如创业公司：CEO 直管所有成员；大型企业：CEO—Team Leader—Member）；预设仅为可选模板。
- **与可观测、记忆、技能**：CEO 与实体共享「实例级」可观测视图（见第七节）；记忆可区分实体私有与实例内共享作用域（见第九节）；技能绑定到实例或实体，由实例/实体配置与工作区加载（见第十一节）。

## 七、Agent 通信

### 7.1 通信范围（概念）

- **单实例内跨实体**：同一实例内，实体与实体、实体与 CEO 之间需要协作与消息传递；平台提供**实例内通信**机制（如消息总线、收件箱、任务派发），供 CEO 分配任务、实体汇报、团队内协作。具体协议与路由在详细设计中定义。
- **实例与管理员之间**：实例**向管理员汇报**（进度、资源申请、异常上报）；管理员**不**向实例开放实例管理能力；汇报内容仅作为信息或审批请求，由管理员（或用户）决策后通过配置或指令反馈。汇报通道与格式在详细设计中定义，与「跨实例 A2A」区分：当前阶段以「汇报 + 用户驱动管理」为主，不依赖对等的跨实例 Agent 互调。
- **跨实例 Agent 互调（可选、后续）**：若未来需要「实例 A 的 CEO 直接向实例 B 的 CEO 发起协作请求」，可在汇报通道之上扩展**跨实例 A2A** 协议与权限控制；本版仅保留概念，不约束实现时机。

### 7.2 模块职责（概念）

- **实例内通信模块**：负责单实例内实体/CEO 间的消息路由、收件箱、订阅与权限校验（如仅 CEO 可创建团队、仅团队成员可访问团队通道）。
- **汇报通道模块**：负责实例→管理员的汇报与资源申请的上报、去重与投递；管理员侧仅消费与展示，不自动执行管理动作。
- **跨实例通信模块（可选）**：若引入 A2A，负责跨实例消息格式、路由、权限与审计；与管理员实例的「全局路由」能力衔接。

## 八、可观测性

### 8.1 分层看板（概念）

为不同角色提供与其职责匹配的**视图**，不约定具体实现形式（Web/API/Telegram 等）。

| 层级 | 看板/视图 | 主要用户 | 典型内容（概念） |
|------|-----------|----------|------------------|
| **L5 用户看板** | 全局摘要与建议 | 用户（自然人） | 集群概览、实例数量与状态、资源与成本摘要、告警与建议、快捷操作入口。 |
| **L4 管理员看板** | 多实例与全局运维 | 管理员实例（用户分身） | 实例列表与状态、重大事件、成本分析、待审批（资源申请等）、实例详情入口。 |
| **L3 CEO 看板** | 单实例内运营 | CEO Agent | 本实例项目/目标列表、待审批事项、团队与实体状态、资源使用、告警。 |
| **L2 团队看板** | 团队内执行 | 团队负责人（若有） | 任务进度、成员状态、团队内知识/记忆摘要。 |
| **L1 实体看板** | 执行与收件 | 实体（Worker） | 当前任务、执行记录、健康状态、收件箱（实例内消息）。 |

数据流概念：执行与状态自下而上聚合（实体→团队→CEO→管理员→用户）；各层仅暴露该角色有权看到的数据；刷新策略与存储由详细设计定义。

### 8.2 模块职责（概念）

- **指标与事件采集**：从 Agent 执行、工具调用、实例/实体状态、资源使用等处采集指标与事件，供聚合与看板消费。
- **分层聚合**：按「实例—实体—团队」等维度聚合，避免原始数据过载；管理员侧可做跨实例聚合。
- **审计与追溯**：关键操作（实例创建/删除、约束变更、跨实例/跨实体通信）可审计记录，与安全与合规需求衔接。

## 九、记忆与上下文管理

### 9.1 记忆作用域（概念）

在「单实例多实体」下，记忆需区分作用域，避免混淆与越权：

- **实体私有（工作记忆）**：仅当前实体可见；用于当前任务上下文、临时结果、未升档的中间信息。
- **团队/项目共享**：同一团队或项目内实体可共享的记忆（若平台支持团队粒度）；由 CEO 或团队负责人决定共享策略。
- **实例级共享**：实例内所有实体（含 CEO）可访问；用于实例目标、公共知识、实例级策略。
- **全局/跨实例（可选）**：若支持跨实例知识共享，由管理员或管理员授权发起，与「汇报」通道分离；本版仅保留概念。

与 ZeroClaw 的差距：当前 ZeroClaw 为单进程单 Memory 后端，无「按实体/团队/实例」的作用域划分；实现时需在 Memory 之上增加作用域与访问控制层，或为不同作用域提供不同存储/命名空间。

### 9.2 自动压缩与长期记忆（概念）

为支持长时间运行与有限上下文窗口，引入**记忆压缩**与**长期记忆**概念：

- **记忆折叠/压缩**：当上下文（如对话轮数、token 估算）超过设定阈值时，将一段历史**压缩为结构化摘要**（如情景摘要、关键决策、工具调用要点），存入长期记忆；当前会话仅保留**滑动窗口**（最近 N 轮）与**关键事件缓存**，以控制上下文长度。触发条件、压缩粒度与摘要格式在详细设计中定义。
- **长期记忆**：压缩后的摘要与显式保存的重要记忆，以可检索形式存储（如向量+关键词）；实体或 CEO 在需要时**按需检索**，而非全量加载。
- **重要性评分与衰减（可选）**：对记忆条目可赋予重要性或衰减策略，用于自动清理低价值内容、保留高价值经验；具体策略在详细设计中定义。

模块职责（概念）：**记忆上下文管理模块**负责窗口、阈值与触发；**记忆压缩模块**负责生成摘要与写入长期存储；**记忆存储与检索**与现有 ZeroClaw Memory 能力对接，并增加作用域与多租户支持。

## 十、故障恢复与健康

### 10.1 实例健康（概念）

- **健康检查**：对每个实例（进程或进程内实例）进行周期性健康检查（如存活、响应、资源占用）；结果汇总到管理员侧或平台层，供看板与恢复策略使用。
- **健康状态**：至少区分正常、异常、不可用等状态；可扩展为更细粒度（如过载、超时、错误率），供告警与自动恢复使用。

### 10.2 检查点与快照（概念）

- **检查点/快照**：在关键节点（如任务阶段完成、定时）对实例或任务状态做**快照**，包括必要的记忆状态、任务队列、分配关系等，以便故障后恢复。快照粒度（实例级/任务级）、存储与保留策略在详细设计中定义。
- **不替代持久化**：检查点是对「运行状态」的补充，不替代配置、实例注册表与记忆后端的正常持久化。

### 10.3 恢复策略（概念）

- **恢复策略**：当实例被判定为异常或不可用时，可根据策略执行**重启**、**从检查点回滚**或**重建实例**（从注册表与配置重新创建）。策略可由管理员配置或由平台默认。
- **恢复执行**：恢复动作由**管理员侧**或**平台/引导层**执行（如重启进程、重新加载快照），不由普通实例自行执行，以避免权限混乱。
- **与可观测的衔接**：健康状态与恢复事件进入可观测与审计，便于用户与管理员了解故障与恢复历史。

## 十一、技能与扩展

### 11.1 技能归属（概念）

- **技能** 以实例或实体为粒度加载与执行：实例级技能对该实例内所有实体（含 CEO）可用；实体级技能仅该实体可用。具体以配置与工作区约定为准（如 ZeroClaw/OpenClaw 的 skills 目录与清单）。
- **兼容性**：技能格式与加载方式与 ZeroClaw/OpenClaw 现有机制兼容（如 SKILL.md、TOML 清单、工作区路径），以便复用现有技能生态与市场；实现时可在兼容层上增加作用域与权限校验。

### 11.2 模块职责（概念）

- **技能加载与解析**：按实例/实体配置解析技能清单，加载技能内容与元数据；不在此版规定具体运行时（如是否 WASM、是否子进程）。
- **技能执行与隔离**：技能在执行时受实例约束与安全策略限制；与「资源与配额」结合，避免单技能耗尽实例资源。
- **技能市场对接**：若对接 OpenClaw 或自建技能市场，仅约定「技能包格式与安装目标（实例/实体）」的概念，具体协议在详细设计中定义。

---

## 十二、资源与配额

### 12.1 资源维度（概念）

- **实例级约束**：由管理员配置，包括 Agent/实体数量上限、调用量（如 API 调用次数/频率）、资源限额（CPU、内存、存储等，若平台支持）；CEO 在实例内分配任务与资源时不得超出这些约束。
- **全局资源**：若存在全局配额（如总 Token、总成本、总实例数），由管理员或平台层管理与校验；实例创建与约束变更时需校验不突破全局上限。

### 12.2 模块职责（概念）

- **配额管理**：存储与校验实例约束与全局配额；在实例创建、约束更新、实体创建时执行校验。
- **使用量采集**：从执行层与 Provider 等采集调用量、资源使用，供看板与配额校验使用。
- **超限处理**：当使用量接近或超过配额时，可拒绝新请求、触发告警或通过汇报向管理员申请放宽；具体策略在详细设计中定义。

## 十三、概念层次与数据流小结

- **用户** → 仅通过受控入口驱动**管理员实例**的实例管理能力；消费 L5 用户看板与告警。
- **管理员实例** → 拥有全局视图、实例管理能力与 L4 看板；接收实例汇报与资源申请；不向实例开放管理能力；可配置全局/实例约束、健康与恢复策略。
- **实例** → 由管理员创建并受约束；内部由 **CEO** 组织实体；可向管理员汇报；具备实例内通信、L3/L2/L1 看板、记忆作用域、技能加载与资源使用；健康状态与检查点上报或由平台采集。
- **CEO** → 实例内目标与团队设计、实体与资源分配；受管理员下发的约束限制；消费 L3 看板、实例内记忆与技能；不拥有实例管理能力。
- **实体** → 实例内执行单元；参与实例内通信、消费 L1 看板与私有/共享记忆、使用实例/实体技能；受实例与 CEO 分配的资源与任务约束。

横向能力线贯穿上述层次：**通信**（实例内 + 实例→管理员）；**可观测**（五层看板 + 审计）；**记忆**（作用域 + 压缩 + 长期检索）；**故障恢复**（健康 + 检查点 + 恢复策略）；**技能**（实例/实体加载与隔离）；**资源**（配额与使用量）。

---

## 十四、与项目定位及企业级服务的对应

- **Civigenesis / MultiClaw**：本概念设计中的三层角色（管理员、实例、CEO+实体）与六条能力线（通信、可观测、记忆、故障恢复、技能、资源）构成 MultiClaw 的**集群与编排抽象**及**企业级服务能力基线**，为超大规模智能体集群与商业化交付提供统一概念基础。
- **Beyond Skyrim 实验**：经济实验、文明实验对应不同实例；实例内公司/社会结构由 CEO 与预设决定；记忆共享与压缩支持长时运行；可观测与健康支持实验运维与复现。
- **商业服务**：实例约束与资源配额与计费、RaaS、私有化部署策略对接；可观测与审计支持 SLA 与合规；故障恢复支持可用性承诺；技能与扩展支持技能市场与定制能力。

## 十五、后续设计依赖
