# MultiClaw / Civigenesis 执行方案（第一版）

> 基于 ZeroClaw 现有实现与《概念设计》的第一版可执行方案，明确改造点、阶段划分与交付物。具体代码级设计在后续文档中展开。

---

## 一、ZeroClaw 控制与初始化现状（简要）

### 1.1 入口与配置

| 位置 | 行为 |
|------|------|
| **`main.rs`** | 解析 CLI → 若 `Onboard` 则走 onboard 分支后 return；否则 `Config::load_or_init().await` 加载配置，再按子命令分发。 |
| **`Config::load_or_init()`** | 解析 `ZEROCLAW_CONFIG_DIR` / `ZEROCLAW_WORKSPACE` 或默认 `~/.zeroclaw`，确定 `zeroclaw_dir` 与 `workspace_dir`；`config_path = zeroclaw_dir.join("config.toml")`。若文件存在则反序列化并解密密钥；否则 `Config::default()` 写回并返回。**单份全局 Config，无 instance_id 概念。** |
| **`config.apply_env_overrides()`** | 环境变量覆盖 `default_provider`、`default_model`、`api_key` 等。 |

### 1.2 初始化流程（Onboard）

| 路径 | 入口 | 行为 |
|------|------|------|
| **交互式** | `Commands::Onboard { interactive: true, ... }` | `onboard::run_wizard(force).await`，多步引导（channel、provider、model、memory 等），写 `config.toml`。 |
| **快速** | `Commands::Onboard { api_key, provider, model, memory, ... }` | `onboard::run_quick_setup(...).await`，非交互写 config。 |
| **仅通道** | `Commands::Onboard { channels_only: true }` | `onboard::run_channels_repair_wizard().await`，只修 channel 配置。 |
| **后置** | 可选 `ZEROCLAW_AUTOSTART_CHANNELS=1` | Onboard 成功后 `channels::start_channels(config).await`。 |

当前 Onboard 只生成**单份** config，对应**单实例**；无「管理员实例」与「实例注册表」的创建逻辑。

### 1.3 运行形态与配置绑定

| 命令 | 实现位置 | 与 Config 的关系 |
|------|----------|------------------|
| **Agent** | `agent::run(config, message, ...)` | 单 config → `Agent::from_config(&config)` 构建单 Agent，跑单次或交互 loop。 |
| **Gateway** | `gateway::run_gateway(&host, port, config)` | 单 config 放入 `AppState`，所有 webhook/chat 共用该 config，无按实例路由。 |
| **Daemon** | `daemon::run(config, host, port)` | 单 config 克隆给 gateway、channels、heartbeat、cron 四个 supervisor；**单进程单 config**，无多实例。 |
| **Service** | `service::handle_command(..., config, init_system)` | 用 `config.config_path` 与 `config.workspace_dir` 生成 systemd/launchd/OpenRC 单元，启动命令为执行 `zeroclaw daemon`（**单实例 daemon**）。 |
| **Channel** | `channels::start_channels(config)` | 单 config，拉取 channel 配置、创建 provider、memory、runtime，启动各 channel 监听；消息进来后走单 agent 处理。 |

结论：当前**无「实例」抽象**，**无管理员与普通实例之分**，**无单实例多实体**；所有控制流均绑定到一份 `Config` 和一个进程（或 daemon 内一组组件）。

## 二、概念设计到实现的差距（Gap）

与《概念设计》对照，需补齐的能力与改造点如下。

| 概念 | 当前 ZeroClaw | 目标 |
|------|----------------|------|
| **管理员实例（Administrer）** | 无 | 默认初始化时创建并启动；唯一具备「实例管理能力」的实例；仅用户可调用管理能力。 |
| **实例（Instance）** | 无；等价于「单进程单 config」 | 显式实例抽象：每实例独立 config、生命周期、约束；由管理员创建/关闭/删除；实例注册表持久化。 |
| **实例管理能力** | 无 | 查看全局实例列表与详情、创建/关闭/删除实例、配置实例约束（Agent 数、调用量、资源）；以工具或 API 形式暴露给管理员 Agent，且仅用户入口可触发。 |
| **实例汇报** | 无 | 普通实例可向管理员汇报进度、资源申请；管理员不向实例开放管理能力。 |
| **单实例多实体** | 无；仅 delegate 工具做临时子任务 | 实例内多 Entity，每实体独立配置；CEO Agent 在约束内创建团队/实体、分配目标与资源。 |
| **CEO Agent** | 无 | 实例内「实例管理员」角色，负责目标设计、团队设计、资源管理；受管理员下发的约束限制。 |
| **引导与启动** | 用户执行 onboard/daemon；无「平台先起、再起管理员」 | 明确「引导层」只启动管理员实例；管理员在初始化流程中创建；service install 安装的是「启动管理员」的单元。 |

## 三、执行阶段划分

以下四阶段可顺序执行，阶段内任务可并行或拆子任务。

---

### 阶段 0：引导与管理员实例创建（Bootstrap）

**目标**：确立「谁先启动」「管理员从哪来」；在默认初始化流程中创建管理员实例与实例注册表；不改变现有单实例 daemon 的默认行为（兼容现有用户）。

**改造与新增**：

| 模块 | 改造内容 |
|------|----------|
| **配置与目录** | 引入「实例根目录」概念：例如 `~/.zeroclaw` 为集群根；其下 `instances/admin/` 为管理员实例的 config + workspace；`instances/<id>/` 为普通实例。或沿用单目录但增加 `instance_id` 解析（如环境变量 / 启动参数）。定义 `InstanceRegistry` 持久化结构（如 `instances.json` 或 SQLite）：实例 id、状态、config 路径、创建时间、约束摘要。 |
| **Onboard** | 默认 Onboard 流程中：若检测到「首次初始化」或「无管理员实例」，则创建管理员实例目录与 config（或标记当前 config 为 admin），并写入实例注册表（仅一条 admin 记录）。若已存在管理员，则保持现有 Onboard 行为（可配置单实例或「加入已有集群」）。 |
| **main.rs** | 支持启动时指定 `--instance admin`（或从环境变量读取）；`Config::load_or_init()` 扩展为可按 instance_id 解析到对应 config 路径；无参数时保持「单实例」行为（加载默认 config，等价于当前）。 |
| **Service** | `zeroclaw service install` 时，若目标为「集群模式」，则生成的单元中启动命令带 `--instance admin`（或等效），保证 OS 启动时只拉起管理员实例进程。单实例模式仍为当前 `zeroclaw daemon`。 |

**交付物**：

- 文档：集群根目录与实例目录约定、实例注册表格式。
- 代码：实例注册表读写、Onboard 时创建 admin 实例与注册表、main 支持 `--instance`、service 支持「安装管理员实例为系统服务」。

**依赖**：无。**出口标准**：执行默认 Onboard 后存在「管理员实例」目录与注册表；`zeroclaw daemon --instance admin` 能使用该 config 启动。

---

### 阶段 1：实例抽象与管理员实例管理能力

**目标**：实现「实例」作为一等公民（独立 config、生命周期）；管理员实例具备「查看全局、查看详情、创建/关闭/删除实例、配置约束」的能力；仅用户可调用这些能力；普通实例可向管理员汇报。

**改造与新增**：

| 模块 | 改造内容 |
|------|----------|
| **Config / 实例** | 每个实例对应一份 config 文件（或已存在目录下的 `config.toml`）；实例约束（agent 数量上限、调用量、资源限额）可落在 config 或注册表扩展字段；加载 config 时绑定 `instance_id`。 |
| **实例生命周期** | 新增模块（如 `instance_manager` 或扩展现有 crate）：创建实例（生成目录、写入默认 config、注册表插入）、关闭实例（停进程、更新状态）、删除实例（删目录与注册表项）；与「进程 spawn」解耦时，关闭/删除仅改状态与文件，实际进程由 daemon 或外部编排器根据注册表管理。 |
| **管理员工具** | 在管理员实例的 Agent 工具集中，新增仅管理员可用的工具（或通过 capability 控制）：如 `instances_list`、`instance_detail`、`instance_create`、`instance_stop`、`instance_delete`、`instance_set_constraints`。实现内调用上述实例生命周期与注册表。 |
| **访问控制** | Gateway 或 Channel 层：识别请求是否来自「用户」（如配对后的会话、指定 channel 的 user_id 白名单）；仅用户触发的会话可调用管理员实例的「实例管理」类工具；其他来源仅能触发非管理能力或汇报。汇报：普通实例通过固定 API 或消息通道向管理员实例发送「进度/资源申请」消息，管理员侧仅处理为「信息」不触发管理动作。 |
| **Daemon（可选本阶段）** | 若 daemon 仍为「单进程单实例」：管理员实例以 `zeroclaw daemon --instance admin` 运行；普通实例由管理员「创建」后，由**外部**脚本或后续编排器按注册表 spawn 独立进程（`zeroclaw daemon --instance <id>`）。或在本阶段只实现「管理员能创建/关闭/删除实例的元数据与目录」，实际进程的启停由 Phase 2 的编排器或人工完成。 |

**交付物**：

- 实例 CRUD 与注册表更新逻辑；实例约束的存储与校验接口。
- 管理员专属工具实现与权限校验逻辑。
- 文档：管理员能力列表、调用边界、汇报协议草案。

**依赖**：阶段 0。**出口标准**：管理员实例可列出实例、查看某实例详情、创建/关闭/删除实例（含目录与注册表）、配置约束；仅用户入口可触发管理能力；普通实例可发送汇报消息。

---

### 阶段 2：单实例多实体与 CEO 机制

**目标**：在单个实例进程内支持多实体（多 Agent 实体）；引入 CEO Agent，负责实例内目标设计、团队设计、资源分配；CEO 仅在管理员下发的约束内操作；平台提供预设组织模式（如创业公司、大型企业），不强制内部拓扑。

**改造与新增**：

| 模块 | 改造内容 |
|------|----------|
| **实体抽象** | 在实例内引入 `EntityId`、`EntityConfig`（或复用/扩展现有 `Config` 的子结构）；每个实体有独立 provider/model、记忆作用域、工具集（可裁剪）；实体列表与配置存储在实例 config 或实例状态文件中。 |
| **CEO Agent** | 实例内固定一个「实例管理员」实体，角色名如 CEO；其工具集包含：创建/删除团队、创建/删除实体、分配目标与资源、查看实例内状态；这些工具在调用时校验「未超出管理员下发的约束」（agent 数量、调用量、资源）。CEO 不暴露「实例管理能力」（不能创建/关闭其他实例）。 |
| **约束校验** | 实例启动时加载管理员配置的约束；CEO 在执行「创建实体/团队」等操作时，检查当前实体数、调用量等是否已达上限；超限则拒绝并可选地通过汇报向管理员申请放宽。 |
| **运行时** | 单进程内：一个「实例」对应一个进程；进程内维护实体池（EntityPool）；每个实体可映射到独立的 Agent 运行上下文（或共享 Provider 池但独立 session）；调度器按请求或回合驱动各实体。具体为「多线程/多 task 每实体」或「单 loop 轮询」由详细设计定。 |
| **预设模式** | 提供 1～2 种预设（如创业公司：CEO 直管所有实体；大型企业：CEO → Team Leader → Member）；预设体现为「实例创建时的模板」或「CEO 可加载的初始团队结构」；不限制 CEO 后续自定义。 |

**交付物**：

- 实体与 CEO 的配置与运行时模型；约束校验点与错误返回。
- CEO 工具实现（团队/实体 CRUD、资源分配、状态查询）。
- 至少一种预设组织模式的配置或模板。
- 文档：单实例多实体架构、CEO 能力与约束、预设使用说明。

**依赖**：阶段 1（实例与约束已存在）。**出口标准**：在一个实例进程内可运行多实体；CEO 能创建团队与实体并在约束内分配任务；预设模式可选。

---

### 阶段 3：集成与生产化

**目标**：Gateway 支持按实例路由（可选）；Daemon 或编排器支持「管理员 + 多实例」的启动与监控；Service 安装与运维流程闭环；可观测性与故障恢复满足生产要求。

**改造与新增**：

| 模块 | 改造内容 |
|------|----------|
| **Gateway** | 若多实例以多进程形式存在：Gateway 可仅挂在管理员实例上，用户请求只达管理员；管理员通过工具或内部 RPC 与各实例通信。或：统一网关根据请求头/参数将 webhook 路由到对应实例（需网关持有实例列表与端口/套接字映射）。本阶段至少实现「管理员实例的 Gateway 可服务用户」；实例侧若需独立入口再扩展。 |
| **Daemon / 编排** | 管理员实例以 daemon 运行；普通实例可由「管理员创建实例」时触发的子进程 spawn（同一台机），或由独立编排器（如 systemd 模板、K8s）按注册表启动。若在进程内模拟多实例（单进程多实例），则 daemon 内需调度多个实例的 gateway/channels/agent 组件，复杂度较高；建议先采用「一实例一进程」与外部编排。 |
| **Service** | `zeroclaw service install` 在集群模式下安装「仅启动管理员实例」的单元；文档说明普通实例由管理员创建后需单独部署或由编排器管理。 |
| **可观测与恢复** | 实例注册表持久化；管理员重启后从注册表恢复实例列表与状态；各实例健康状态可由实例自报或管理员轮询；关闭/崩溃实例的清理与重试策略。 |

**交付物**：

- Gateway 与实例的访问路径约定（含可选按实例路由方案）。
- Daemon/编排与 Service 的部署文档与推荐拓扑。
- 实例注册表恢复与健康检查逻辑；运维手册草案。

**依赖**：阶段 1、阶段 2。**出口标准**：用户通过管理员实例的 Gateway/Channel 使用集群；管理员可管理多实例；实例可多实体运行；服务安装与重启后管理员与注册表正确恢复。

## 四、ZeroClaw 模块与阶段对照表

| 模块/文件 | 阶段 0 | 阶段 1 | 阶段 2 | 阶段 3 |

| **instance_manager（新）** | — | 实例 CRUD、约束写入 | — | 与编排/进程启停对接 |
| **tools（管理员工具，新或扩展现有）** | — | instances_list/detail/create/stop/delete/set_constraints | — | — |
| **gateway/** | 无 | 用户 vs 非用户识别，管理能力保护 | 无 | 按实例路由（可选） |
| **daemon** | 无 | 无或仅 admin 单实例 | 无 | 多实例启动策略或文档 |

| **channels** | 无 | 汇报目标指向管理员 | 无 | 无 |

## 五、依赖与里程碑（建议）

- **M0**：阶段 0 完成 — 引导与管理员实例可创建、注册表存在、`--instance admin` 可启动。
- **M1**：阶段 1 完成 — 管理员具备实例管理能力且仅用户可调用；实例可创建/关闭/删除并配置约束；汇报通道可用。
- **M2**：阶段 2 完成 — 单实例内多实体与 CEO 可用；至少一种预设模式；约束生效。
- **M3**：阶段 3 完成 — Gateway/Daemon/Service 与可观测性满足「集群可部署、可运维」的初版要求。

后续可在此基础上迭代：身份/支付/信用、A2A 协议、多机编排、更丰富预设与 CEO 策略等。
