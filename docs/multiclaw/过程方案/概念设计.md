# MultiClaw / Civigenesis 概念设计（第一版）

> 面向超大规模智能体集群的编排与实验基础设施 — 仅做概念性抽象设计，不涉及具体代码与接口定义。

---

## 一、文档目的与范围

- **目的**：明确「管理员实例 + 多实例 + 单实例多实体」的抽象层次、职责边界与协作关系，为后续详细设计与实现提供概念基线。
- **范围**：全局架构、管理员实例能力、实例生命周期与约束、单实例内多实体与 CEO 机制、预设组织模式；不包含协议细节、数据结构与代码级设计。

---

## 二、总体架构概览

系统分为三层抽象：

1. **全局层**：一个**管理员实例（Administrer）**，作为用户的数字分身与集群控制平面，负责查看全局状态、管理所有实例。
2. **实例层**：多个**普通实例（Instance）**，每个实例对应一类实验或业务（如经济实验、文明实验），拥有独立配置与生命周期，由管理员创建、配置、关闭与删除。
3. **实体层**：在**单个实例内部**，多个**智能体实体（Entity）** 并行存在，由该实例的**实例管理员 Agent（如 CEO）** 在管理员授予的约束内，自主设计团队结构与目标，实现公司/实验目标。

关系简述：

- 仅**用户**可调用管理员实例的**实例管理能力**（创建/关闭/删除实例、配置约束等）。
- 其他实例**不能**调用管理员能力，只能向管理员**汇报**（如进度、资源申请）。
- 实例内部由 **CEO Agent** 在授权范围内自由设计组织与目标；具体实体间的协作模式不做平台级约束，可由 CEO 或预设模板决定。

## 三、管理员实例（Administrer）

### 3.1 角色与定位

- **身份**：用户的数字分身 + 整个集群的全局管理员。
- **启动**：在系统**默认初始化流程**中创建并启动，与「平台/引导层」启动后即存在；不依赖其他实例。
- **可见性**：可查看全局状态及所有实例的汇总与详情，是集群的单一控制与观测入口（面向用户）。

### 3.2 能力范围（仅概念列表）

管理员实例具备以下能力，统称为**实例管理能力**：

| 能力类别 | 含义（概念级） |
|----------|----------------|
| **查看全局实例基本信息** | 获取当前集群中所有实例的列表及摘要信息（如实例 ID、类型、状态、创建时间、约束摘要等）。 |
| **查看具体实例详情** | 针对指定实例，查看其配置、运行状态、资源使用、以及内部 Agent 运行记录等详细视图。 |
| **创建实例** | 按用户或策略要求，创建新的实例（如经济实验实例、文明实验实例），并赋予初始配置与约束。 |
| **关闭实例** | 对指定实例执行「关闭」操作，使其停止运行并释放资源；关闭后实例可从列表中移除或标记为已关闭。 |
| **删除实例** | 在关闭基础上，删除实例的元数据与持久化数据（若存在），使其不可恢复。 |
| **配置实例约束** | 为实例设置或更新约束，例如：Agent 数量上限、调用量/配额、资源限额（CPU/内存/存储等）、以及其它策略性限制。 |

上述能力在实现时可能拆分为若干「管理类工具」或 API，此处不展开。

### 3.3 调用与访问控制

- **谁可以调用管理员的实例管理能力**：**仅用户**（通过受控的入口，如已认证的会话、指定 Channel、或管理端 API）。其他实例、其他 Agent **不得**直接调用管理员的「创建/关闭/删除实例、配置约束」等能力。
- **其他实例与管理员的关系**：  
  - 其他实例**不能**调用管理员能力。  
  - 其他实例**可以**向管理员**汇报**，例如：汇报进度、发起资源申请、上报异常或状态。汇报行为不包含「创建/删除/配置实例」等管理动作，仅包含信息上报与请求类交互。

这一边界保证：只有用户（或用户明确授权的代理）能改变集群拓扑与约束，避免实例或恶意 Agent 越权管理。

## 四、实例层（Instance）

### 4.1 定义与职责

- **实例**：对应一个独立实验或业务单元（如一个经济实验、一个文明演化实验）。每个实例拥有**单独配置**，包括但不限于：目标描述、资源约束、允许的 Agent 数量与调用量（由管理员配置）、以及实例内组织与实体的默认策略（若使用预设模板）。
- **生命周期**：由管理员创建而诞生，可被管理员关闭或删除；实例运行期间，内部由该实例的「实例管理员 Agent（CEO）」负责目标与团队设计，平台不强制其实体拓扑。

### 4.2 与管理员的关系

- 实例的**创建、关闭、删除、约束配置**均由管理员在用户授权下执行。
- 实例可向管理员**汇报**（进度、资源申请等），不执行管理动作。
- 管理员可**查看**实例的基本信息与详情（配置、运行状态、内部 Agent 运行记录等），用于全局观测与运维。

### 4.3 约束（由管理员配置）

管理员可为每个实例配置约束，例如：

- **Agent 数量**：该实例内允许存在的实体（Agent）数量上限。
- **调用量**：如对模型调用、外部 API 调用等的频率或总量限制。
- **资源限额**：CPU、内存、存储等资源上限（若平台支持）。
- 其它策略性约束（如允许的模型集合、网络策略等），可在后续设计中补充。

实例内部的 CEO 与实体必须在上述约束内运作；超出约束的申请需通过汇报向管理员或用户请求放宽（由产品与实现决定）。

## 五、单实例多实体与 CEO 机制

### 5.1 基本假设

- 在一个**实例内部**，可并行存在**多个智能体实体**，每个实体可有**独立配置**（如角色、目标、可用工具、模型等）。
- 实例内设有一个**实例管理员 Agent**，在本方案中称为 **CEO Agent**（或同义词，如「实例协调者」）。CEO 负责该实例的**全局目标设计、团队设计、资源分配与协调**，并在管理员（Administrer）授予的约束范围内运作。

### 5.2 CEO 的职责与权限（概念）

- **目标设计**：根据实例的使命（如「经营一家创业公司」「运行一段文明演化」），制定或细化全局目标与阶段目标。
- **团队设计**：在管理员配置的约束内（如 Agent 数量上限、资源限额），**自由创建团队、分配角色、设定汇报关系**，以落实目标。CEO 不超出管理员授予的配额与限额。
- **资源管理**：在实例内部对资源（如任务分配、调用预算、优先级）进行分配与调度；若需更多资源或放宽约束，则通过**向管理员汇报/申请**的方式提出，由用户或管理员决策。

CEO 的「创建团队 / 创建实体 / 分配角色」等动作，在实现上可能对应「在实例内创建子实体、绑定配置、建立汇报链」等机制，具体由后续设计定义。

### 5.3 实体与协作模式的不约束原则

- **平台不固定**单实例内部的实体间拓扑与协作模式。具体采用「扁平团队」「层级汇报」「矩阵式」等，由 **CEO Agent 根据实例目标自行设计**。
- 平台可提供**预设模板**，供用户或 CEO 选用，以降低配置成本；但模板仅为可选快捷方式，不限制 CEO 自定义组织形态。

### 5.4 预设组织模式（示例）

以下为可提供的**预设模式**示例，用于常见场景的快速搭建；具体命名与参数以产品为准。

| 预设名称 | 概念描述 | 典型用途 |
|----------|----------|----------|
| **创业公司模式** | CEO 直接管理所有成员，无中间团队层；成员向 CEO 汇报，CEO 分配任务与目标。 | 小规模协作、快速迭代、单层决策。 |
| **大型企业模式** | CEO 创建多个团队，每个团队有团队负责人（Team Leader）；Team Leader 创建并管理成员，成员向 Leader 汇报，Leader 向 CEO 汇报。 | 多团队、多层级、分工明确的大规模仿真。 |
| （其它） | 可根据需求增加更多预设（如项目制、矩阵式等）。 | 由产品与实验设计决定。 |

预设仅影响实例的初始结构或推荐结构；运行过程中 CEO 仍可在约束内调整团队与成员（若能力设计允许）。

## 六、概念层次与数据流小结

- **用户** → 仅用户可驱动**管理员实例**的实例管理能力（查看全局、创建/关闭/删除实例、配置约束）。
- **管理员实例** → 拥有全局视图与管理权；接收其他实例的**汇报**（进度、资源申请等）；不向其他实例开放管理能力。
- **实例** → 由管理员创建并受约束；内部由 **CEO Agent** 在授权范围内设计目标与团队；实例可向管理员汇报，不可调用管理员能力。
- **CEO Agent** → 实例内部唯一具备「团队设计、实体创建、资源分配」等权限的 Agent；在管理员给定的 Agent 数量、调用量、资源限额内自由组织。
- **实体（Entity）** → 实例内的执行单元，配置与关系由 CEO（或预设模板）决定；平台不约束具体协作模式。

---

## 七、与项目定位的对应关系

- **Civigenesis / MultiClaw**：本概念设计中的「管理员实例 + 多实例 + 单实例多实体」构成 MultiClaw 的**集群与编排抽象**，为「超大规模智能体集群」提供分层管理、隔离与协作基础。
- **Beyond Skyrim 实验**：经济实验、文明实验等可对应为**不同实例**；每个实验内部的公司/社会结构由该实例的 **CEO Agent** 与预设或自定义模式决定，支撑「自负盈亏的智能体公司」与「文明演化」等场景。
- **商业服务**：管理员对实例的约束配置、资源限额与调用量等，可与后续的计费、RaaS、企业私有化策略对接；具体在详细设计中展开。

## 八、后续设计依赖

本概念设计完成后，可进一步产出（由后续任务拆分）：

- 管理员实例与实例的**生命周期与状态机**（创建、运行、关闭、删除等状态及转换条件）。
- **实例管理能力**的细粒度列表与调用边界（含权限与审计）。
- **单实例多实体**的运行时模型：CEO 与实体的创建/销毁、配置存储、约束校验点、以及向管理员汇报的协议或接口形式。
- **预设组织模式**的参数量与可选配置项，以及与 CEO 决策的衔接方式。
- 与身份、支付、信用等商业能力的衔接点（仅概念或接口级）。
