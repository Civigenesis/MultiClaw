# MultiClaw / Civigenesis 执行方案 3.0

> 在《执行方案 2.0》基础上，纳入七项优化：实体 Provider 配置策略、ClawHub 技能自主安装与安全扫描、对话/群聊驱动、扩展通信机制、可观测性 Web 架构、内置组织类型、业务级故障恢复。**本版对每项优化进行专门分析，并给出可落地的设计规格。**

---

## 一、文档目的与 3.0 变更摘要

### 1.1 目的

- 整合用户提出的七项优化需求；
- 对每项需求进行分析，并在设计上做出取舍或提出讨论点；
- 形成与 2.0 兼容且可执行的 3.0 设计基线。

### 1.2 3.0 新增与变更

| 优化点 | 2.0 状态 | 3.0 变更 |
|--------|----------|----------|
| 1. 实体 Provider 配置 | 实体可配 provider/model | 明确「统一注册 + 实体覆盖」的混合策略，并给出分析结论 |
| 2. 技能 ClawHub 自主安装 | 仅格式兼容与迁移 | 增加 ClawHub 浏览/搜索/安装、作用域（实例/全局）、安装前平台扫描 + CEO 风险评估 |
| 3. 对话/群聊驱动 | 仅任务驱动 | 增加 conversation 消息类型、1:1/群聊/频道、对话驱动与随机社交 |
| 4. 通信机制扩展 | 汇报为主 | 增加 CEO 资源申请、管理员→CEO 下发消息、实例→全局信息/知识共享（管理员审批） |
| 5. 可观测性 | 分层概念 | 设计可观测架构、数据模型、API、Web 页面与图表（甘特图、看板等） |
| 6. 内置组织类型 | 仅 startup/enterprise | 增加五类完整设计：初创、集团、群聊头脑风暴、无组织自由讨论、项目制 |
| 7. 故障恢复 | 实例级检查点 | 扩展为「业务级」：运行中记忆、任务、对话、收件箱等无缝衔接 |

## 二、优化点 1：实体 Provider 配置策略

### 2.1 分析

| 方案 | 优点 | 缺点 |
|------|------|------|
| **统一注册到 config** | API Key 集中管理；配置简单；易于成本控制 | CEO 与实体无法按需选模型；无法做成本/能力分层 |
| **每实体独立配置** | 灵活：CEO 用强模型、工人用轻量模型 | 配置复杂；API Key 易重复或泄露风险增加 |
| **混合（推荐）** | 实例级默认 + 实体可覆盖 provider/model；Key 统一存 config | 需明确继承与覆盖规则 |

### 2.2 设计结论：混合策略

- **实例级默认**：`config.default_provider`、`config.default_model`、`config.api_key` 等与现有 ZeroClaw 一致，作为所有实体的默认值。
- **实体级覆盖**：`entity.provider`、`entity.model` 可选；若指定，该实体使用覆盖值；否则继承实例默认。
- **API Key 与凭证**：**一律存放在实例 config**，不在实体配置中重复；实体仅指定 provider/model 名称，凭证从 `model_providers` 或 `api_key` 解析。
- **配置结构（设计级）**：

```toml
# 实例 config
[instance]
default_provider = "openrouter"   # 可选，继承 config 顶层
default_model = "anthropic/claude-sonnet-4"

[instance.entities.ceo]
provider = "anthropic"            # 覆盖：CEO 用 Anthropic
model = "claude-sonnet-4-20250514"

[instance.entities.worker-1]
# 不指定则使用 instance 默认

[instance.entities.code-reviewer]
provider = "openai"               # 覆盖：代码评审用 OpenAI
model = "gpt-4o"
```

- **与 ZeroClaw**：现有 `config.model_providers`、`config.api_key` 保持不变；实体加载时按 `entity.provider` 解析到对应凭证，若无则用 `default_provider` 与 `api_key`。

## 三、优化点 2：ClawHub 技能自主安装与安全扫描

### 3.1 能力设计

- **浏览与搜索**：CEO 或实体（需权限）可调用 `clawhub_search`、`clawhub_browse` 工具，对接 ClawHub/OpenClaw 技能库 API；返回技能列表、描述、评分、依赖等。
- **自主安装**：选定技能后可发起 `clawhub_install`；安装目标默认为**实例级**（`scope: instance`）；若申请安装到**全局**（`scope: global`），需**管理员审批**。
- **作用域规则**：
  - `instance`：安装到 `instances/<id>/workspace/skills/<name>/`；仅该实例可用。
  - `global`：安装到 `cluster_root/shared/skills/<name>/`；需管理员审批后生效；所有实例可选用。

### 3.2 安装前安全扫描（设计级）

**建议：平台强制扫描 + CEO/Agent 可选风险评估**

| 层级 | 职责 | 说明 |
|------|------|------|
| **平台扫描（强制）** | 静态分析、沙箱检测 | 检测路径穿越、危险命令、可疑网络请求、已知恶意模式；**不通过则禁止安装**。 |
| **CEO/Agent 风险评估（可选）** | 业务与上下文判断 | CEO 或实体对技能做「业务风险」评估（如是否适合当前实例）；平台扫描通过后，可由 CEO 二次确认；**不替代平台扫描**。 |

- **讨论点**：若完全由 CEO/Agent 自行扫描，其若被投毒或判断失误，可能放行恶意技能。因此**平台扫描为必选**，CEO 评估为补充。
- **流程**：`clawhub_install` 请求 → 平台拉取技能包 → 平台扫描 → 不通过则拒绝；通过且 `scope=global` → 进入管理员审批队列；通过且 `scope=instance` → 可直接安装，或由实例配置决定是否需 CEO 确认。

### 3.3 工具与 API（概念）

- **工具**：`clawhub_search`、`clawhub_browse`、`clawhub_install`；安装时传入 `skill_id`、`scope`（instance/global）。
- **管理员工具**：`skill_approve_global`、`skill_reject_global` 用于审批全局安装。
- **ClawHub 对接**：定义与 ClawHub 的协议（如 REST 搜索、包下载 URL）；实现时可为适配层，支持 OpenClaw 技能市场或自建 ClawHub。

## 四、优化点 3：对话与群聊驱动

### 4.1 需求与场景

- **任务驱动**：已有（CEO 分配任务 → 实体执行）。
- **对话驱动**：Agent 间的非任务型交互，如：
  - 文明涌现：随机社交、偶遇、闲聊；
  - 公司场景：实体主动发起讨论、头脑风暴、群聊。

### 4.2 设计：消息类型与通道

#### 4.2.1 消息类型扩展

| 类型 | 用途 | 特点 |
|------|------|------|
| `task` | 任务分配、执行指令 | 有完成状态、可追踪 |
| `conversation` | 对话、讨论、社交 | 无强制完成态；可多轮 |
| `broadcast` | 公告、通知 | 一对多，无回复链 |

#### 4.2.2 通道类型

| 通道 | 参与者 | 发起方式 |
|------|--------|----------|
| **1:1** | 两个实体 | 任一方发起，对方同意或自动加入 |
| **群聊（Group）** | 多个实体 | 创建时指定成员或主题；可邀请加入 |
| **频道（Channel）** | 实例内订阅者 | 创建后实体订阅；适合广播型讨论 |
| **随机社交** | 实例内任意实体 | 按配置概率随机匹配或加入某群 |

#### 4.2.3 对话驱动机制

- **主动发起**：实体在空闲时可选择「发起对话」或「发起群聊」；或由配置的 `conversation_initiate_probability`、`conversation_idle_trigger_secs` 触发。
- **随机社交**：文明涌现场景下，可配置「每 N 分钟随机选 2～3 个实体进行简短对话」；对话内容写入共享记忆（可选）。
- **与任务驱动协同**：实体收件箱同时接收 `task` 与 `conversation`；调度器可配置优先级（如任务优先，或轮询混合）。

#### 4.2.4 实现要点（设计级）

- **ConversationBus**：与 MessageBus 并列或扩展；支持 `create_conversation`、`invite`、`post`、`leave` 等操作。
- **Conversation 状态**：`conversation_id`、`type`（1:1/group/channel）、`participants`、`messages`（可持久化到 Memory 或专用存储）。
- **调度**：实体在 `poll_idle` 时，除检查任务外，还可检查「被邀请的对话」「可发起的随机社交」；按配置决定是否参与。

### 4.3 配置项（设计级）

| 配置项 | 说明 | 默认 |
|--------|------|------|
| `instance.conversation.enabled` | 是否启用对话/群聊 | true |
| `instance.conversation.initiate_interval_secs` | 实体发起对话的最小间隔 | 300 |
| `instance.conversation.random_social_interval_secs` | 随机社交触发间隔（文明涌现） | 600 |
| `instance.conversation.max_participants` | 群聊最大参与人数 | 10 |

## 五、优化点 4：通信机制扩展

### 5.1 2.0 已有 vs 4 扩展

| 方向 | 2.0 | 3.0 扩展 |
|------|-----|----------|
| 实例→管理员 | 汇报（progress、alert） | + **资源申请**（resource_request） |
| 管理员→实例 | （无） | + **管理员→CEO 下发消息** |
| 实例→全局 | （无） | + **信息/讨论/知识共享**（需管理员审批） |

### 5.2 资源申请（CEO→管理员）

- **格式**：`ResourceRequestMessage { instance_id, from_entity: ceo, kind: resource_request, request: { agent_quota_increase, call_quota_increase, ... }, reason, timestamp }`。
- **流程**：CEO 调用 `request_resources` 工具 → 写入汇报通道 → 管理员侧展示在「待审批」；用户/管理员审批后，调用 `instance_set_constraints` 等更新约束；结果通过「管理员→CEO 消息」反馈。

### 5.3 管理员→CEO 消息

- **格式**：`AdminToCeoMessage { instance_id, content, kind: instruction|approval|rejection|info, ref_request_id?, timestamp }`。
- **传输**：管理员实例通过实例 Gateway 的专用端点（如 `POST /api/admin-message`）或共享存储投递；实例侧轮询或 WebSocket 拉取。
- **用途**：下发指令、审批结果、全局公告、告警通知。

### 5.4 实例→全局信息/知识共享（需管理员审批）

- **格式**：`GlobalShareRequest { from_instance_id, kind: info|discussion|knowledge, title, content, target: all_instances|specific, timestamp }`。
- **流程**：实例（CEO 或授权实体）发起 `request_global_share` → 进入管理员审批队列；管理员审批通过后，写入**全局共享存储**（如 `cluster_root/global/knowledge/` 或专用表）；其他实例可读取或订阅。
- **审批**：管理员可 `approve_global_share`、`reject_global_share`；审批结果通知发起实例。

### 5.5 模块职责小结

- **汇报通道**：扩展支持 `resource_request`。
- **管理员下发通道**：新增，负责 admin→CEO 消息的投递与确认。
- **全局共享**：新增存储与订阅机制；审批通过后写入，实例可查询 `list_global_knowledge`、`get_global_share`。

## 六、优化点 5：可观测性架构与 Web 展示

### 6.1 架构概览

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          Observability Stack                             │
├─────────────────────────────────────────────────────────────────────────┤
│  Data Sources:  Agent events | Tool calls | Memory | Health | Reports    │
│       ↓                                                                  │
│  Aggregation:   instance_id, entity_id, team_id, timestamps              │
│       ↓                                                                  │
│  Store:         Events DB / Time-series / Checkpoints                    │
│       ↓                                                                  │
│  API Layer:     REST / WebSocket (real-time)                             │
│       ↓                                                                  │
│  Web UI:        User/Admin | CEO | Team | Entity  Views                  │
└─────────────────────────────────────────────────────────────────────────┘
```

### 6.2 各角色视图与数据要求

| 角色 | 主视图 | 数据内容 | 图表类型 |
|------|--------|----------|----------|
| **用户/管理员** | 全局仪表盘 | 资源使用（Token、成本、实例数）；各实例基本信息（状态、实体数）；**待审批**（资源申请、全局共享）；**汇报内容**（按实例聚合） | 资源折线图、实例状态卡片、审批列表、汇报时间线 |
| **CEO** | 实例运营台 | 实例内资源使用；团队列表与状态；任务列表与进度；实体列表与健康 | 任务甘特图、团队看板、实体状态表 |
| **团队** | 任务看板 | 本团队任务、进度、阻塞项、成员负载 | 看板（To Do / In Progress / Done）、甘特图、燃尽图 |
| **实体** | 执行台 | 当前任务、历史执行记录（每轮 turn、工具调用、结果） | 任务详情、执行时间线、日志列表 |

### 6.3 关键图表设计

| 图表 | 用途 | 数据维度 |
|------|------|----------|
| **甘特图** | 任务时间线、依赖 | task_id、start、end、assignee、status、predecessors |
| **看板** | 任务状态流转 | 列=状态；卡片=任务；可拖拽 |
| **资源图** | Token/成本趋势 | 时间、instance_id、entity_id、value |
| **汇报时间线** | 汇报与审批流 | 时间、instance_id、kind、content、status |
| **执行记录表** | 实体每轮执行 | entity_id、turn、tool_calls、output、duration |

### 6.4 Web 页面结构（设计级）

| 页面 | 路由 | 内容 |
|------|------|------|
| 登录/认证 | `/login` | 用户认证；根据角色跳转 |
| 用户/管理员仪表盘 | `/dashboard` | 资源总览、实例列表、待审批、汇报聚合 |
| 实例详情 | `/instances/:id` | 实例配置、健康、汇报历史、快捷操作 |
| CEO 视图 | `/instances/:id/ceo` | 团队、任务、实体、资源（需 CEO 会话或模拟） |
| 团队看板 | `/instances/:id/teams/:tid` | 团队任务看板、甘特图 |
| 实体视图 | `/instances/:id/entities/:eid` | 实体任务、执行记录、收件箱摘要 |
| 审批中心 | `/approvals` | 资源申请、全局共享的审批列表 |

- **技术选型**：前端可复用 ZeroClaw 现有 web 技术栈；图表可用 ECharts、Gantt 库等；实时刷新可用 WebSocket 或轮询。

### 6.5 数据采集与存储

- **事件类型**：`task_created`、`task_assigned`、`task_completed`、`turn_start`、`turn_end`、`tool_call`、`report`、`approval_request`、`health_update` 等。
- **存储**：事件表（append-only）；聚合表（按 instance/entity/team 预聚合）；保留策略可配置（如 7 天原始、30 天聚合）。

## 七、优化点 6：内置组织类型

### 7.1 五类预设

| 类型 | 英文标识 | 适用场景 |
|------|----------|----------|
| 初创公司 | `startup` | 小团队、CEO 直管、扁平 |
| 集团公司 | `enterprise` | 多层级、CEO—TeamLeader—Member |
| 群聊头脑风暴 | `brainstorm` | 无固定层级、讨论驱动 |
| 无组织自由讨论 | `freeform` | 文明涌现、可自由组队或 1:1 |
| 项目制 | `project` | 项目为中心、跨职能团队 |

### 7.2 各类完整设计

#### 7.2.1 初创公司（startup）

- **结构**：CEO 直管所有实体；无 Team 中间层。
- **通信**：任务直接分配到实体；可选 1:1 或群聊（CEO 发起）。
- **配置**：`preset: startup`；`instance.entities` 仅 CEO + 若干 worker；无 `teams`。
- **适用**：小公司模拟、快速验证。

#### 7.2.2 集团公司（enterprise）

- **结构**：CEO → TeamLeader（多个）→ Member；每个 Team 有 leader 与 members。
- **通信**：任务可分配给 Team 或实体；Team 内可群聊；跨 Team 需经过 Leader 或 CEO。
- **配置**：`preset: enterprise`；`instance.teams` 定义团队；`instance.entities` 含 role（ceo/team_leader/member）与 `team_id`。
- **适用**：大型组织模拟、层级管理。

#### 7.2.3 群聊头脑风暴（brainstorm）

- **结构**：无固定层级；所有实体平等；以「讨论组」为单位。
- **通信**：以 conversation 为主；可创建多个头脑风暴组；成员可自由加入/离开；任务可后验从讨论中提炼。
- **配置**：`preset: brainstorm`；`instance.entities` 无 role 层级；`conversation` 权重高于 `task`。
- **适用**：创意讨论、问题研讨。

#### 7.2.4 无组织自由讨论（freeform）

- **结构**：无预设层级；实体可自由发起 1:1、群聊；可动态形成临时「组织」（如两人结盟、小组）。
- **通信**：完全由对话与社交驱动；可配置随机社交频率；实体可 `form_group`、`leave_group`。
- **配置**：`preset: freeform`；`instance.conversation` 全开；`random_social` 启用；可选 `allow_organic_teams`。
- **适用**：文明涌现、社会模拟、Emergence 实验。

#### 7.2.5 项目制（project）

- **结构**：以项目为核心；每个项目有 project_owner、members；项目可跨 Team；实体可属于多项目。
- **通信**：任务绑定到项目；项目内群聊；项目看板独立。
- **配置**：`preset: project`；`instance.projects` 定义项目；`instance.entities` 含 `project_ids`；任务有 `project_id`。
- **适用**：多项目并行、敏捷开发模拟。

### 7.3 预设与配置映射

| preset | teams | projects | conversation_weight | 特殊 |
|--------|-------|----------|---------------------|------|
| startup | 无 | 无 | 低 | CEO 直管 |
| enterprise | 有 | 可选 | 中 | 层级 |
| brainstorm | 无（讨论组） | 无 | 高 | 讨论为主 |
| freeform | 无/动态 | 无 | 高 | 随机社交、有机组队 |
| project | 可选 | 有 | 中 | 项目看板 |

## 八、优化点 7：业务级故障恢复

### 8.1 2.0 与 3.0 差异

| 维度 | 2.0 | 3.0 |
|------|-----|-----|
| 检查点粒度 | 实例级、任务队列、分配关系 | + **运行中记忆**、**对话缓冲区**、**当前 turn 状态** |
| 恢复目标 | 实例进程重启 | **业务无缝衔接**：任务、对话、记忆从断点恢复 |
| 持久化频率 | 定时或阶段完成 | + **关键状态实时持久化**（或高频率 checkpoint） |

### 8.2 需持久化的业务状态

| 状态 | 说明 | 恢复时用途 |
|------|------|------------|
| **任务队列** | 待分配、进行中、阻塞 | 恢复后继续分配与执行 |
| **任务执行进度** | 当前 turn、工具调用中、部分结果 | 从断点继续或安全重试 |
| **收件箱** | 每实体的未读消息 | 恢复后继续消费 |
| **对话缓冲区** | 进行中的 1:1/群聊消息 | 恢复后继续对话 |
| **记忆上下文** | 当前加载到会话的 Memory 摘要 | 恢复后避免重复加载或丢失上下文 |
| **实体运行态** | 当前实体、当前 prompt、history 快照 | 恢复后接续 turn |
| **CEO 循环状态** | 上次分析时间、待处理决策 | 恢复后按周期继续 |

### 8.3 检查点设计（扩展）

- **全量检查点**：包含上述所有状态；周期可配置（如每 5 分钟或每 N 个 turn）。
- **增量检查点**：仅持久化自上次全量后的变更（如新任务、新消息）；恢复时合并增量到最近全量。
- **关键点检查点**：在「任务完成」「对话轮次结束」等节点触发；减少恢复时的回放量。

### 8.4 恢复流程（设计级）

1. 实例进程启动时，检测是否存在未完成的 checkpoint；
2. 若有，加载最近全量 + 增量（如有）；
3. 恢复 EntityPool、MessageBus、ConversationBus、任务队列、收件箱；
4. 恢复各实体的「当前 turn」状态：若在工具执行中，根据策略决定重试或跳过；
5. 恢复 CEO 循环、Cron、Heartbeat 的调度状态；
6. 继续运行；可观测中记录「从 checkpoint 恢复」事件。

### 8.5 与 ZeroClaw 的衔接

- **Memory 持久化**：现有 Memory 后端已持久化；需确保 checkpoint 与 Memory 的写入顺序一致，或 checkpoint 仅记录「引用」而非完整记忆内容。
- **Daemon 状态**：现有 `daemon_state.json` 可扩展为包含 `last_checkpoint_path`、`recovery_mode` 等字段。

## 十、讨论与建议

### 10.1 建议采纳的设计

- **Provider 混合策略**：兼顾灵活与安全。
- **平台强制扫描 + CEO 可选评估**：避免完全依赖 Agent 导致的安全盲区。
- **对话/群聊机制**：满足文明涌现与公司场景的社交需求。
- **通信扩展**：资源申请、管理员下发、全局共享均有明确场景。
- **可观测 Web 架构**：分层清晰，可分期实现。
- **五类预设**：覆盖主要组织形态。
- **业务级恢复**：支撑长时间运行与可靠性。

### 10.2 可讨论点

- **技能扫描**：若坚持「仅 CEO/Agent 自行扫描」，需接受其被投毒或误判的风险；建议保留平台强制扫描作为底线。
- **对话频率**：随机社交与发起间隔的默认值可根据实际负载调优；高频率可能增加成本。
- **全局共享存储**：可先采用文件系统（`cluster_root/global/`），后续扩展为数据库或向量库。
