# MultiClaw / Civigenesis 执行方案 2.0

> 基于《概念设计 2.0》与《执行方案》第一版，结合 ZeroClaw 实现机制的深入分析，形成更详细的执行级设计。**本版不写具体代码，但给出多实例、单实例多实体、技能兼容、Agent 主动性及其他机制的完整设计，供实现时直接落地。**

---

## 一、文档目的与 ZeroClaw 现状摘要

### 1.1 目的

- 将《概念设计 2.0》中的各模块转化为可执行的设计规格；
- 与 ZeroClaw 现有实现对齐，避免与配置、Onboard、Daemon、Gateway、Agent、Skills、Memory 等模块冲突；
- 为多实例、单实例多实体、技能兼容、Agent 主动性、通信、可观测、记忆、故障恢复等提供统一的设计基线。

### 1.2 ZeroClaw 现状（实现级摘要）

| 模块 | 当前实现要点 |
|------|--------------|
| **Config** | 单份 `Config`；`config_path = zeroclaw_dir/config.toml`；`workspace_dir` 与 zeroclaw_dir 关联；由 `ZEROCLAW_CONFIG_DIR` / `ZEROCLAW_WORKSPACE` 或默认 `~/.zeroclaw` 解析；无 `instance_id` 概念。 |
| **Onboard** | `onboard::run_wizard()` / `run_quick_setup()` 写 `config.toml` 到 `zeroclaw_dir`；scaffold `workspace_dir` 下 `sessions`、`memory`、`state`、`cron`、`skills` 等子目录；无实例注册表、无 admin 实例。 |
| **main.rs** | Onboard 分支后 return；否则 `Config::load_or_init().await` 加载单份 config；无 `--instance` 参数；按子命令分发。 |
| **Daemon** | `daemon::run(config, host, port)`；单 config 克隆给 gateway、channels、heartbeat、scheduler 四个 supervisor；gateway 绑定 `host:port`（默认 42617）；Ctrl+C 终止全部。 |
| **Gateway** | `run_gateway(host, port, config)`；单 config 放入 `AppState`；所有 webhook/chat 共用该 config；无按实例路由。 |
| **Agent** | `agent::run(config, message, provider, model, temp, peripheral, interactive)`：`message.is_some()` 时单次 turn 后返回；`interactive` 时 stdin 循环，用户输入驱动，/quit 退出；**无内置「持续主动运行」循环**。 |
| **Cron** | `cron::scheduler::run(config)` 定时轮询 `due_jobs()`；JobType::Shell 执行命令；JobType::Agent 调用 `agent::run(config, Some(msg), ...)`；单 config。 |
| **Heartbeat** | 定时触发 `agent::run(config, Some(prompt), ..., false)`；单次调用，无持续循环。 |
| **Skills** | `load_skills_with_config(workspace_dir, config)` 从 `workspace_dir/skills/<name>/` 加载；优先 `SKILL.toml`，否则 `SKILL.md`；可选 open-skills repo；无实例/实体作用域。 |
| **Memory** | 单 backend（sqlite/lucid/markdown/none）；`workspace_dir` 下的存储；无按实体/实例的作用域划分。 |
| **Service** | `zeroclaw service install` 生成 systemd/launchd 单元，启动 `zeroclaw daemon`；单实例。 |

## 二、多实例与单实例多实体的详细设计

### 2.1 需深入的设计点（分析结论）

在引入「管理员实例 + 普通实例 + 单实例多实体」时，需明确以下设计点并保持与 ZeroClaw 的兼容：

1. **集群根与实例目录**：当前 ZeroClaw 为 `~/.zeroclaw` 或 `ZEROCLAW_CONFIG_DIR` 下的单 config；多实例需引入「集群根」与「实例子目录」约定。
2. **端口分配**：当前 Daemon/Gateway 单端口；多实例若一实例一进程，每实例需独立端口。
3. **启动机制**：当前 `zeroclaw daemon` 单进程单 config；需支持 `--instance <id>` 与实例注册表驱动的进程启停。
4. **文件目录**：当前 `zeroclaw_dir` 下有 `config.toml`、`workspace`、`daemon_state.json` 等；多实例需隔离各实例的 config、workspace、state。
5. **实例注册表**：需持久化实例 id、状态、config 路径、端口、约束、创建时间等。
6. **单实例多实体**：在实例进程内维护 Entity 列表与 CEO；每实体独立 provider/model、记忆作用域、工具集。

### 2.2 集群根与实例目录设计

#### 2.2.1 目录布局

沿用《执行方案》第一版的思路，与 ZeroClaw 现有约定兼容：

```
<cluster_root>/                      # 集群根，如 ~/.zeroclaw 或 ZEROCLAW_CLUSTER_ROOT
├── instances.json                   # 实例注册表（新增）
├── instances/
│   ├── admin/                       # 管理员实例（instance_id = "admin"）
│   │   ├── config.toml              # 该实例的 config
│   │   ├── daemon_state.json        # 该实例的 daemon 状态（若 daemon 运行）
│   │   └── workspace/               # 该实例的 workspace
│   │       ├── sessions/
│   │       ├── memory/
│   │       ├── state/
│   │       ├── cron/
│   │       └── skills/
│   └── <instance_id>/               # 普通实例，如 exp-economy、exp-civilization
│       ├── config.toml
│       ├── daemon_state.json
│       └── workspace/
│           └── ...
```

- **集群根**：由 `ZEROCLAW_CLUSTER_ROOT` 或 `ZEROCLAW_CONFIG_DIR`（扩展语义）指定；默认 `~/.zeroclaw`。
- **向后兼容**：若集群根下**无** `instances.json` 且**无** `instances/` 目录，则视为「单实例模式」：`config.toml` 与 `workspace` 直接在集群根下（即当前 ZeroClaw 行为）；`instance_id` 可视为 `"default"` 或隐式单实例。
- **管理员实例**：`instances/admin/` 固定存在；Onboard 在「集群模式」下创建该目录与注册表。

#### 2.2.2 实例注册表 `instances.json` 结构（设计级）

| 字段 | 类型 | 说明 |
|------|------|------|
| `version` | u32 | 注册表格式版本 |
| `instances` | array | 实例列表 |
| `instances[].id` | string | 实例 id，如 `admin`、`exp-economy` |
| `instances[].role` | enum | `admin` \| `normal` |
| `instances[].status` | enum | `created` \| `running` \| `stopped` \| `deleted` |
| `instances[].config_path` | string | config.toml 的绝对或相对路径 |
| `instances[].workspace_path` | string | workspace 路径 |
| `instances[].gateway_port` | u16 | 该实例 Gateway 监听端口 |
| `instances[].created_at` | string | ISO8601 |
| `instances[].constraints` | object | agent_max、call_quota、resource_limits 等 |
| `instances[].pid` | int? | 运行中进程 PID（可选，用于健康检查） |

- 注册表由**管理员实例**或**平台引导层**写入；普通实例不直接修改。
- 端口分配策略：管理员实例使用 config 中 `gateway.port`（或 42617）；普通实例由管理员在创建时分配，从可用端口池中选取（如 42618+），并写入注册表。

### 2.3 端口分配设计

| 角色 | 端口来源 | 约定 |
|------|----------|------|
| **管理员实例** | `instances/admin/config.toml` 中的 `[gateway]` port | 默认 42617；可由用户配置 |
| **普通实例** | 管理员创建时从端口池分配 | 端口池：42618～43000（可配置）；写入注册表 `gateway_port` |
| **冲突处理** | 分配前检查端口占用 | 若端口被占用，选取下一个可用端口；持久化到注册表 |

- 端口分配模块职责：提供 `allocate_port(used_ports: &[u16], pool_start: u16, pool_end: u16) -> u16` 的语义；实现时可用 `std::net::TcpListener::bind("127.0.0.1:0")` 获取随机可用端口，或顺序扫描。
- 多机部署时：各机器独立端口空间；跨实例通信若需要，由「实例地址」= `host:port` 表示。

### 2.4 启动机制设计

#### 2.4.1 main.rs 与 Config 加载

- **CLI 扩展**：增加 `--instance <id>`（或 `-i <id>`）；若指定，则加载该实例的 config；若不指定且存在集群根 + 注册表，则可根据环境变量或默认策略选择实例（如 `ZEROCLAW_INSTANCE=admin`）。
- **Config 加载路径**：
  - 若 `--instance admin`：`cluster_root/instances/admin/config.toml`；
  - 若 `--instance <id>`：`cluster_root/instances/<id>/config.toml`；
  - 若无 `--instance` 且无注册表：保持当前 `Config::load_or_init()` 行为（`zeroclaw_dir/config.toml`）。
- **Onboard 扩展**：在「首次集群初始化」时，创建 `instances/admin/`、写入 `instances.json`；Onboard 完成后，用户可执行 `zeroclaw daemon --instance admin` 启动管理员实例。

#### 2.4.2 Daemon 启动

- **管理员实例**：`zeroclaw daemon --instance admin`（或 `zeroclaw daemon` 且默认 instance=admin）；行为与当前 Daemon 一致，但 config 来自 `instances/admin/config.toml`。
- **普通实例**：由管理员通过「实例管理能力」创建后，**进程启停**有两种实现路径：
  - **路径 A（推荐）**：管理员调用 `instance_create` 时，仅创建目录、config、注册表；实际进程由**外部编排器**（systemd 模板、K8s、自定义脚本）根据注册表执行 `zeroclaw daemon --instance <id>`；管理员调用 `instance_stop` 时，更新注册表状态，编排器轮询或 watch 注册表后停止进程。
  - **路径 B**：管理员进程内 spawn 子进程，`Command::new("zeroclaw").args(["daemon", "--instance", id])`；需管理子进程句柄、日志、退出码。
- 本方案建议**路径 A**，以减少管理员进程的职责，便于多机扩展。

#### 2.4.3 Service 安装

- `zeroclaw service install` 在集群模式下：生成的单元中，ExecStart 为 `zeroclaw daemon --instance admin`；即 OS 启动时仅拉起管理员实例。
- 普通实例由管理员创建后，需单独部署或由编排器管理；文档说明「普通实例的 systemd 模板示例」或 K8s manifest 示例。

### 2.5 单实例多实体的管理方式

#### 2.5.1 实体与 CEO 的配置

- **实例 config 扩展**：在现有 `Config` 上增加可选 `[instance]` 或 `[entities]` 段（或复用 `config.agents` 的扩展）：
  - `instance.ceo`：CEO 的 agent 配置（provider、model、system_prompt 等）；
  - `instance.entities`：实体列表，每实体有 `entity_id`、`role`、`provider`、`model`、`tools`、`memory_scope` 等；
  - `instance.preset`：预设组织模式，如 `startup`（CEO 直管所有）、`enterprise`（CEO—TeamLeader—Member）。
- **CEO**：实例内固定一个「实例管理员」实体；其 `entity_id` 可为 `ceo`；工具集包含 `create_team`、`create_entity`、`assign_task`、`instance_status` 等；不暴露 `instances_list`、`instance_create` 等「实例管理能力」。

#### 2.5.2 实体与记忆/技能的作用域

- **记忆**：每实体可有 `memory_scope`：`private`（仅自己）、`team:<team_id>`（团队内共享）、`instance`（实例内共享）；Memory 后端需支持 key 前缀，如 `entity:<id>:`、`team:<id>:`、`instance:`。
- **技能**：实例级技能 = `workspace/skills/` 下加载，对所有实体可见；实体级技能 = 可在实体配置中指定 `skills: [a, b]`，仅该实体加载对应技能；加载路径仍为 `workspace/skills/<name>/`，通过白名单过滤。

#### 2.5.3 运行时模型（设计级）

- **单进程内**：一个实例进程 = 一个 `InstanceRuntime`；内含 `EntityPool`：`HashMap<EntityId, EntityRuntime>`；每 `EntityRuntime` 对应一个 Agent 运行上下文（history、session、tools）。
- **调度**：由「主动性机制」（见第四节）驱动：CEO 周期性分析目标、分配任务；实体根据收件箱执行任务；或由 Cron/Heartbeat 等外部触发。
- **约束校验**：CEO 在执行 `create_entity` 时，检查当前实体数是否已达 `constraints.agent_max`；超限则拒绝并可选汇报管理员。

### 2.6 文件目录小结

| 路径 | 说明 |
|------|------|
| `<cluster_root>/instances.json` | 实例注册表 |
| `<cluster_root>/instances/admin/` | 管理员实例目录 |
| `<cluster_root>/instances/<id>/` | 普通实例目录 |
| `<cluster_root>/instances/<id>/config.toml` | 实例 config |
| `<cluster_root>/instances/<id>/workspace/` | 实例 workspace（sessions、memory、state、cron、skills 等） |
| `<cluster_root>/instances/<id>/daemon_state.json` | 该实例 daemon 的 health 快照（若 daemon 运行） |
| `<cluster_root>/instances/<id>/entities.json` | 实例内实体列表与配置（可选；或存于 config 或 state） |

- 单实例模式（向后兼容）：`<cluster_root>/config.toml`、`<cluster_root>/workspace/` 直接存在，无 `instances/` 子目录。

## 三、OpenClaw 技能兼容的详细机制

### 3.1 ZeroClaw 与 OpenClaw 技能格式对比

| 维度 | ZeroClaw | OpenClaw（参考） |
|------|----------|------------------|
| **技能根目录** | `workspace/skills/<name>/` | 类似 `workspace/skills/` 或插件目录 |
| **清单格式** | `SKILL.toml` 或 `SKILL.md` | `SKILL.md`、可能另有 manifest |
| **SKILL.toml** | `[skill]` name、description、version、author、tags；`[[tools]]` name、description、kind、command、args | 需对齐字段 |
| **SKILL.md** | 纯 Markdown，name 取自目录名，description 取首行非空 | 多为说明型，Agent 按需读取 |
| **工具定义** | tools 数组中 shell/http/script 等 | 可能有 parameters 规范 |
| **open-skills** | `open-skills` repo，`skills/<name>/SKILL.md` 结构 | 社区技能仓库，格式兼容 |

### 3.2 兼容层设计

#### 3.2.1 格式统一

- **主格式**：以 ZeroClaw 的 `SKILL.toml` + `SKILL.md` 为主；`SKILL.toml` 优先，若无则解析 `SKILL.md`。
- **字段映射**：若 OpenClaw 有额外字段（如 `parameters`、`category`、`implementation`），在加载时映射到 ZeroClaw 的 `Skill` 结构；未知字段可存入 `metadata` 或忽略。
- **技能包格式**：若从 OpenClaw 技能市场下载的为 zip/tar，解压后应得到 `skills/<name>/` 目录，且内含 `SKILL.md` 或 `SKILL.toml`；安装时拷贝到 `workspace/skills/<name>/`，与 ZeroClaw 现有 `skills install` 行为一致。

#### 3.2.2 加载路径与优先级

| 来源 | 路径 | 优先级 |
|------|------|--------|
| 实例 workspace | `instances/<id>/workspace/skills/<name>/` | 1（最高） |
| open-skills | `open_skills_dir/skills/<name>/` 或 config 指定 | 2 |
| 全局（可选） | `cluster_root/shared/skills/<name>/` | 3 |

- 同名技能：workspace 覆盖 open-skills；实体级 `skills` 白名单进一步过滤。

#### 3.2.3 与实例/实体的作用域

- **实例级**：实例 config 中可指定 `skills.enabled: [a, b]` 或 `skills.disabled: [x]`；加载时过滤。
- **实体级**：实体配置中 `skills: [c, d]` 表示仅加载 c、d；若无指定，则继承实例级全部。
- **管理员实例**：可加载「实例管理」相关技能（如 `instances_list` 工具所在技能）；普通实例不加载。

#### 3.2.4 OpenClaw 迁移与桥接

- **迁移**：若有 OpenClaw 的 `~/.openclaw/workspace/skills/` 目录，`zeroclaw migrate openclaw --source ~/.openclaw` 可将技能目录拷贝到 ZeroClaw workspace，并确保 SKILL 格式兼容。
- **桥接**：若 OpenClaw 技能包含 `package.json` 或其它 manifest，解析后生成等效的 `SKILL.toml`；无法解析的则保留 `SKILL.md`，按 ZeroClaw 的 `load_skill_md` 逻辑加载。

### 3.3 技能执行与隔离（设计级）

- 技能中的 `tools` 若为 `kind: shell`，执行时受 `autonomy`、`security` 策略限制（与当前 ZeroClaw 一致）。
- 实例/实体级配额：单次工具调用的资源限制可与实例约束关联；超限时拒绝执行并记入可观测。

## 四、Agent 主动性机制

### 4.1 需求背景

- 场景：公司持续赚钱、文明涌现观察等，需要 Agent **持续主动运行**，而非仅响应用户输入。
- 当前 ZeroClaw：Agent 为**被动**——单次 message 或 interactive stdin 驱动；Cron/Heartbeat 为定时触发，但非「Agent 自驱循环」。

### 4.2 主动性机制设计

#### 4.2.1 CEO 主动循环

- **模式**：CEO 作为实例内「目标与任务分配中枢」，周期性运行「分析 → 决策 → 分配任务」循环。
- **触发**：
  - **定时**：例如每 N 分钟（可配置）执行一次 CEO turn；实现上可由实例内轻量调度器或复用 Cron（`JobType::Agent`，目标是 CEO）。
  - **事件**：实体完成关键任务后向 CEO 汇报，触发 CEO 下一轮分析（需实例内通信）。
- **CEO 单次 turn 内容**：输入 = 实例目标 + 团队状态 + 待办 + 汇报摘要；输出 = 新任务分配、团队调整建议、向管理员的汇报草案。
- **与 Cron 的关系**：可新增 `JobType::Ceo`，专门驱动 CEO 的周期性分析；或 `JobType::Agent` 的 `target: ceo` 表示调度 CEO 而非默认单 Agent。

#### 4.2.2 实体主动循环

- **模式**：每个实体可有「待办队列」与「自主执行循环」：当队列非空时，自动取任务执行；队列空时，可主动「巡逻」（如检查收件箱、实例共享记忆、定期自检）。
- **触发**：
  - **任务驱动**：CEO 分配任务进入实体收件箱；实体调度器轮询收件箱，取任务执行。
  - **定时巡逻**：实体可配置 `idle_interval_secs`，空闲时每 N 秒执行一次「自检/巡逻」prompt，如「检查是否有新任务、共享记忆更新、需汇报的事项」。
- **实现形态**：在实例进程内，每实体一个 `EntityLoop` task；循环内：`recv(task)` 或 `poll_idle()` → `agent::run(config, Some(prompt), ..., false)` → 结果写入共享状态/汇报通道。
- **与约束**：实体的主动调用计入实例的 `call_quota`；超限时暂停主动循环，并通过汇报申请放宽。

#### 4.2.3 与现有 Cron / Heartbeat 的集成

- **Cron**：保持 `JobType::Agent` 与 `JobType::Shell`；扩展 `JobType::Agent` 支持 `target: ceo | <entity_id>`，指定调度 CEO 或某实体；若无 target，则调度「默认单 Agent」（当前行为）。
- **Heartbeat**：可配置 `heartbeat.target: ceo` 或 `heartbeat.target: default`，分别驱动 CEO 或默认 Agent 执行心跳任务。
- **统一调度器**：实例内可有一个 `ProactiveScheduler`，负责 CEO 周期、实体空闲巡逻、Cron/Heartbeat 的协调；避免重复触发与资源争用。

### 4.3 主动性配置（设计级）

| 配置项 | 说明 | 默认 |
|--------|------|------|
| `instance.proactive.ceo_interval_secs` | CEO 主动分析间隔（秒） | 300 |
| `instance.proactive.entity_idle_interval_secs` | 实体空闲巡逻间隔 | 60 |
| `instance.proactive.ceo_enabled` | 是否启用 CEO 主动循环 | true |
| `instance.proactive.entity_autonomous` | 实体是否可主动执行队列 | true |
| `instance.proactive.max_concurrent_turns` | 实例内并发 turn 上限 | 3 |

- 超限与退避：当 `call_quota` 接近或达到上限时，主动循环暂停或降频；恢复策略可配置（如管理员放宽后自动恢复）。

## 五、其他机制详细设计

### 5.1 实例内通信

#### 5.1.1 消息总线与收件箱（概念）

- **收件箱**：每实体一个 `inbox`；CEO 或其它实体可向某实体发送 `TaskMessage`；格式可简化为 `{ from, to, kind: task|report|request, payload }`。
- **路由**：实例内 `MessageBus` 根据 `to` 路由到对应实体的 inbox；`to: ceo` 表示 CEO 收件箱；`to: team:<id>` 表示组播到该团队所有实体。
- **与 ZeroClaw 的集成**：当前 Agent 无 inbox；可新增 `tools::inbox_send`、`tools::inbox_recv` 供 CEO/实体调用；或由平台层在每轮 turn 前自动注入「未读收件」到 prompt。

#### 5.1.2 汇报通道（实例 → 管理员）

- **格式**：`ReportMessage { instance_id, kind: progress|resource_request|alert, payload, timestamp }`。
- **传输**：普通实例通过 HTTP POST 到管理员实例的 Gateway 专用端点（如 `/api/report`），或写入共享存储（管理员轮询）；管理员侧仅记录与展示，不自动执行管理动作。
- **认证**：汇报需携带实例凭证（如 instance_token），由管理员校验。

### 5.2 可观测性

#### 5.2.1 分层看板数据源（设计级）

| 层级 | 数据来源 | 聚合维度 |
|------|----------|----------|
| L5 用户 | 管理员实例的全局视图 | 实例数、状态、资源摘要、告警 |
| L4 管理员 | 实例注册表、汇报队列、各实例健康 | instance_id |
| L3 CEO | 实例内任务、实体状态、资源使用 | entity_id, team_id |
| L2 团队 | 团队内任务与成员 | team_id |
| L1 实体 | 实体当前任务、收件箱、执行记录 | entity_id |

- **采集**：复用 ZeroClaw 的 `observability`、`health`、`runtime_trace`；扩展为支持 `instance_id`、`entity_id` 维度；事件写入现有 trace 存储或新增聚合存储。
- **刷新**：各层看板按需轮询或 WebSocket 推送；刷新间隔可配置。

#### 5.2.2 审计

- **范围**：实例创建/删除、约束变更、CEO 的 create_entity/create_team、跨实体/跨实例消息。
- **存储**：append-only 日志或数据库；与现有 `sop/audit` 等能力衔接。

### 5.3 记忆与上下文

#### 5.3.1 作用域实现（设计级）

- **Key 前缀**：`entity:<id>:`、`team:<id>:`、`instance:`；Memory 后端（如 sqlite）的 key 采用该前缀，实现逻辑隔离。
- **访问控制**：读/写时校验调用者 entity_id 与 scope；`instance` 作用域对所有实例内实体可读。
- **与 ZeroClaw**：现有 Memory 接口扩展 `scope` 参数；底层 key 拼接 scope 前缀。

#### 5.3.2 记忆压缩（设计级）

- **触发**：当 `history` 长度或 token 估算超过阈值（如 `agent.max_history_messages` 的 80%）时触发压缩。
- **流程**：选取一段历史，调用 LLM 生成摘要；摘要写入长期记忆；当前窗口保留最近 N 轮 + 摘要引用。
- **长期存储**：与现有 Memory 后端集成；摘要作为新条目写入，category 如 `compressed`。
- **检索**：按需检索时，可查询 `compressed` 类别与实体/实例 scope。

### 5.4 故障恢复与健康

#### 5.4.1 实例健康检查

- **方式**：管理员定期请求各实例的 `/health` 或读取实例自报的 `daemon_state.json`；或实例向管理员汇报心跳。
- **状态**：`healthy`、`degraded`、`unhealthy`、`unknown`。
- **与 ZeroClaw**：现有 `health::snapshot_json()`、`daemon_state.json` 可扩展 `instance_id` 维度；多实例时每实例一份 state 文件。

#### 5.4.2 检查点与快照

- **粒度**：实例级快照 = 关键内存状态（任务队列、分配关系）+ 配置摘要；任务级快照 = 单任务中间状态（可选）。
- **存储**：`instances/<id>/checkpoints/<timestamp>.json`；保留策略（如最近 N 个或最近 7 天）。
- **恢复**：管理员执行「从检查点恢复」时，将快照加载回实例；若实例进程已退出，则需重启进程并传入快照路径。

#### 5.4.3 恢复策略

- **重启**：管理员调用 `instance_restart`；编排器或管理员进程发送终止信号后重新 spawn。
- **回滚**：加载最近可用检查点；需确保 Memory 与检查点一致（或接受最终一致性）。
- **重建**：删除实例目录后重新 `instance_create`；适用于不可恢复的损坏。

### 5.5 资源与配额

#### 5.5.1 配额校验点

| 校验点 | 时机 | 动作 |
|--------|------|------|
| 实例创建 | `instance_create` | 检查全局实例数、端口池 |
| 实体创建 | CEO `create_entity` | 检查 `constraints.agent_max` |
| 每轮 turn | Agent 调用前 | 检查 `call_quota`、`cost_per_day` |
| 工具执行 | 工具执行前 | 检查 `max_actions_per_hour`（与现有 autonomy 一致） |

- **超限**：拒绝操作并返回错误；可选触发汇报「资源申请」。

#### 5.5.2 使用量采集

- 从 `observability`、Provider 回调、cost 模块采集 token、成本、调用次数；按 `instance_id`、`entity_id` 聚合；供看板与配额校验使用。
