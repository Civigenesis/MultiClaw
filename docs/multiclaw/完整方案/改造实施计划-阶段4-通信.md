## 一、资源申请（实例 → 管理员）

### 1.1 行为

| 项目 | 说明 |
|------|------|
| **场景** | 实例内实体/CEO 在 call_quota、cost_per_day、agent_max 等即将或已超限时，可发起 resource_request |
| **传输** | 与汇报共用通道：`POST /api/report`，body 中 `type: "resource_request"`，payload 含 resource_type、current_usage、requested 等 |
| **处理** | 管理员侧存储到待审批或日志；可选：管理员通过 admin-message 回复批准/拒绝，或由人工处理 |

### 1.2 代码修改

| 位置 | 说明 |
|------|------|
| `src/gateway/api.rs` | 解析 report body 的 type，若为 resource_request 则写入 `instances/admin/workspace/state/resource_requests.jsonl` 或合并到 reports 存储 |
| **单元测试** | `gateway::api::tests::report_resource_request_persists` |

### 1.3 实例内触发

| 位置 | 说明 |
|------|------|
| `src/agent/loop_.rs` 或 cost/tracker | 每轮 turn 前/后检查配额；超限或接近时可选调用「发送 report(resource_request)」的 internal 逻辑（或通过 CEO 工具 report_resource_request 供 CEO 调用） |
| **单元测试** | `cost::tracker::tests::emits_resource_request_when_near_limit`（若实现自动发送） |

## 二、实例 → 全局共享（GlobalShareRequest）

### 2.1 流程

1. 实例（或 CEO）发起 `request_global_share`：内容类型 info/discussion/knowledge，target=all_instances|specific，payload。
2. 请求写入管理员侧「待审批列表」。
3. 管理员执行 `approve_global_share` / `reject_global_share`（CLI 或 API）。
4. 批准后：内容写入 `cluster_root/global/knowledge/<id>.json`（或单文件追加）；可选通知目标实例。

### 2.2 数据结构

| 项目 | 说明 |
|------|------|
| **GlobalShareRequest** | `{ id, source_instance_id, content_type, target, payload, created_at, status: pending | approved | rejected }` |
| **存储** | 待审批：`instances/admin/workspace/state/global_share_pending.json`（数组或 jsonl） |
| **已批准** | `cluster_root/global/knowledge/<uuid>.json` 或 `knowledge.jsonl` |

### 2.3 代码修改

| 位置 | 说明 |
|------|------|
| **新工具或 API** | 实例侧：`request_global_share` 工具（CEO 或授权实体可调）；内部发 `POST /api/global-share-request`（或复用 report 的 type=global_share_request） |
| `src/gateway/api.rs` | `POST /api/global-share-request`：校验来源实例，写入 pending；`POST /api/approve-global-share`、`POST /api/reject-global-share`：校验管理员，更新 status 并写入 global/knowledge 或仅移除 pending |
| **单元测试** | `gateway::api::tests::global_share_request_stored_as_pending`，`approve_writes_to_global_knowledge`，`reject_removes_from_pending` |

## 三、管理员 → 实例（AdminToCeoMessage）

- 阶段 1 已实现 `POST /api/admin-message` 并持久化到目标实例的 `admin_messages`。
- 阶段 3 CEO 循环中：每次 CEO tick 前读取该队列，注入到 CEO 的 system 或 context（instruction、approval、rejection、info）。

- **单元测试**：`proactive_scheduler::tests::ceo_receives_admin_messages_before_tick` 或 `gateway::api::tests::admin_message_consumed_by_ceo`（需 mock 或集成）。
