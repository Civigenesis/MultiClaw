## 二、通用约定

### 2.1 向后兼容

- **单实例模式**：无 `instances.json` 且无 `instances/` → 行为与当前一致，`config.toml` / workspace 在集群根（或 `~/.multiclaw`）。
- **集群模式**：存在 `instances/` 且存在 `instances.json` → 多实例、管理员、注册表生效。

### 2.2 测试要求（每模块必达）

- 新增/修改的**核心逻辑**须有**单元测试**（`#[cfg(test)] mod tests` 或 `tests/*.rs`）。
- 测试命名：`<模块>_<行为>_<预期>`（如 `instance_registry_load_returns_empty_when_missing`）。
- 每阶段合并前：`cargo test` 全绿，必要时补充集成测试（如 `tests/config_persistence.rs` 风格）。

### 2.3 代码与配置契约

- 新配置项在 `src/config/schema.rs` 中定义并加 `#[serde(default)]` 或明确默认值；文档更新 `docs/config-reference.md`。
- 新 CLI 子命令在 `src/main.rs` 与 `src/lib.rs` 的 Command 枚举中声明，并在对应模块实现。

## 三、阶段 0：引导与管理员实例

**目标**：集群根、`instances/admin`、`instances.json`、`--instance`、Onboard 可创建 admin。  
**里程碑 M0**：`multiclaw daemon --instance admin` 能正常启动。

### 3.1 目录与数据结构

#### 3.1.1 集群根解析

| 项目 | 说明 |
|------|------|
| **新增** | 集群根路径：`MULTICLAW_CLUSTER_ROOT` 或 `~/.multiclaw` |
| **位置** | `src/config/schema.rs`：新增 `cluster_root()` 或复用/扩展 `default_config_dir()` 逻辑；若存在 `instances.json` 则集群根为该文件所在目录 |
| **单实例判定** | 无 `instances.json` 且无 `instances/` 目录 → 单实例模式，config 路径 = 集群根内 `config.toml`（或现有 `active_workspace` 逻辑） |

**代码修改要点**：

- `src/config/schema.rs`：
  - 增加 `fn cluster_root() -> Result<PathBuf>`：先读 `MULTICLAW_CLUSTER_ROOT`，否则 `default_config_dir()`。
  - 增加 `fn is_cluster_mode(cluster_root: &Path) -> bool`：`cluster_root.join("instances.json").exists() || cluster_root.join("instances").is_dir()`。
  - `resolve_runtime_config_dirs` 或等价逻辑：若集群模式且存在 `--instance`/`MULTICLAW_INSTANCE`，则 `config_path = cluster_root/instances/<id>/config.toml`，`workspace_dir = cluster_root/instances/<id>/workspace`；否则保持现有单实例解析。

**单元测试**：

- `config::schema::tests::cluster_root_uses_env_when_set`
- `config::schema::tests::is_cluster_mode_false_without_instances_json`
- `config::schema::tests::is_cluster_mode_true_with_instances_dir`

#### 3.1.2 实例注册表 `instances.json`

| 项目 | 说明 |
|------|------|
| **新文件** | `src/instance_registry.rs`（或 `src/config/instance_registry.rs`） |
| **结构** | 见架构文档 3.2：`version: u32`, `instances: Vec<InstanceEntry>`，每项含 `id`, `role`, `status`, `config_path`, `workspace_path`, `gateway_port`, `created_at`, `constraints`, `preset`（可选） |
| **操作** | `load( path ) -> Result<InstanceRegistry>`, `save( &self, path ) -> Result<()>`, `get( &self, id ) -> Option<&InstanceEntry>`, `add( &mut self, entry )`, `update_status( &mut self, id, status )` |

**代码修改要点**：

- 定义 `InstanceEntry`、`InstanceRole`、`InstanceStatus`（及可选 `InstanceConstraints`、`preset` 字符串）。
- 实现 `InstanceRegistry { version, instances }` 的 serde 序列化/反序列化，与架构 3.2 字段一致。
- 路径：`cluster_root.join("instances.json")`。

**单元测试**：

- `instance_registry_load_returns_default_when_file_missing`
- `instance_registry_save_and_load_roundtrip`
- `instance_registry_get_returns_entry_by_id`
- `instance_registry_add_and_update_status`

### 3.2 CLI `--instance`

| 项目 | 说明 |
|------|------|
| **位置** | `src/main.rs` |
| **修改** | 在顶层 `Opt` 或等效结构中增加 `--instance <ID>`（Optional<String>）；若未提供则读 `MULTICLAW_INSTANCE`；单实例模式下忽略 |
| **传递** | 子命令（daemon、gateway、onboard、status 等）需要实例上下文时，从 Opt 传入；Config 加载前根据 instance 与集群模式解析 config_path / workspace_dir |

**代码修改要点**：

- `main.rs`：解析 `--instance`，写入 `Opt` 或通过 `clap` 的 `global(true)` 下传。
- 在调用 `Config::load_or_init`（或等价）处：若集群模式且指定了 instance，则使用 `cluster_root/instances/<id>/config.toml`；否则保持现有逻辑。

**单元测试**：

- 可通过 `tests/config_persistence.rs` 或新建 `tests/instance_cli.rs`：运行 `multiclaw status --instance admin` 在预置集群目录下返回成功（或预期错误）。

### 3.3 Onboard 创建 admin 实例

| 项目 | 说明 |
|------|------|
| **位置** | `src/onboard/wizard.rs`（及可能的 `onboard/mod.rs`） |
| **行为** | 在集群模式下，Onboard 可创建「管理员实例」：创建 `instances/admin/` 目录，写入 `config.toml`、`workspace/` 子目录，并在 `instances.json` 中注册一条 `id=admin`、`role=admin`、`status=created`（或 `running`）、`gateway_port=42617` 等 |
| **交互** | 可增加「是否创建集群 / 创建 admin 实例」的向导步骤；若已存在 `instances.json` 且已有 admin，可跳过或仅更新 config |

**代码修改要点**：

- 在 onboarding 流程中：若检测到集群模式或用户选择「创建集群」，则调用 instance_registry 的 `add`，写入 `instances/admin` 目录并生成最小 `config.toml`。
- 确保 `instances.json` 与 `instances/admin/config.toml` 同步写入。

**单元测试**：

- `onboard::tests::create_admin_instance_writes_registry_and_config`
- 可选：集成测试「首次 onboard 后存在 instances/admin 且 instances.json 含 admin」

### 3.4 Daemon 按实例启动

| 项目 | 说明 |
|------|------|
| **位置** | `src/main.rs`（daemon 子命令）、`src/daemon/mod.rs` |
| **修改** | daemon 子命令接收 `--instance <id>`（或从 Opt 读取）；加载对应实例的 Config（见 3.1.1）；`daemon::run(config, host, port)` 不变，仅 config 来源按实例变化 |
| **Service** | `multiclaw service install` 时，ExecStart 写入 `multiclaw daemon --instance admin`（或可配置）；见 `src/service/mod.rs` |

**代码修改要点**：

- `main.rs`：daemon 分支中根据 `--instance` 与集群模式解析 config_path，再 `Config::load_or_init`。
- `service/mod.rs`：安装 systemd/launchd 单元时，若为集群模式默认 `--instance admin`。

**单元测试**：

- `daemon::tests::run_accepts_instance_scoped_config`（可用 mock config 或 temp dir 提供 instances/admin 结构）

### 3.5 阶段 0 交付清单与验收

| 交付物 | 验证方式 |
|--------|----------|
| 集群根 + 单实例/集群判定 | 单元测试 `config::schema` |
| `instances.json` 与 InstanceRegistry | 单元测试 `instance_registry` |
| CLI `--instance` | 集成/CLI 测试 |
| Onboard 创建 admin | 单元测试 + 手动/集成 |
| `multiclaw daemon --instance admin` 启动 | 本地运行 + `cargo test` |

## 四、阶段 1：实例抽象与管理

**目标**：实例 CRUD、端口分配、管理员工具、汇报通道、管理员→CEO 消息。  
**里程碑 M1**：管理员可 CRUD 实例；汇报与 admin→CEO 可用。

### 4.1 实例 CRUD 与端口分配

| 项目 | 说明 |
|------|------|
| **新模块** | `src/instance_manager.rs`（或 `src/instance/manager.rs`） |
| **职责** | 创建实例（分配端口 42618+）、删除实例、列表、启动/停止（更新 status）；端口池从配置或常量读取，分配后写入 InstanceRegistry |
| **依赖** | `InstanceRegistry`、`cluster_root`、Config 中可选 `[instance_manager]` 的 `port_pool_start` |

**代码修改要点**：

- `instance_create(id, role, preset?)`：创建 `instances/<id>/`、最小 `config.toml`、`workspace/`，分配端口，写入注册表。
- `instance_delete(id)`：可选只做软删除（status=deleted）或移除目录；端口回收到池。
- `instance_list()`：从注册表读取并返回。
- 端口分配：维护「已分配端口」集合（可从当前注册表推导），分配时取最小可用 ≥ 42618。

**单元测试**：

- `instance_manager_create_allocates_port_and_writes_registry`
- `instance_manager_delete_frees_port`
- `instance_manager_list_returns_registry_entries`

### 4.2 管理员工具（CLI）

| 项目 | 说明 |
|------|------|
| **位置** | `src/main.rs`、可选 `src/instance/mod.rs` |
| **子命令** | `multiclaw instance create <id> [--preset ...]`, `multiclaw instance list`, `multiclaw instance delete <id>`, `multiclaw instance status <id>`（可选） |
| **权限** | 仅当当前为「管理员实例」或显式 `--admin` 时允许 create/delete（可由 cluster 角色或 token 决定；首版可简化为：在 admin 实例上下文下允许） |

**代码修改要点**：

- 在 `main.rs` 增加 `InstanceCommands` 子命令，委托给 `instance_manager`。
- 创建/删除前检查集群模式及权限（若实现简单权限：当前加载的 config 是否为 `instances/admin`）。

**单元测试**：

- `instance_cli_list_parses`；集成测试：在 temp 集群根下 `instance create` 后 `instance list` 可见。

### 4.3 汇报通道（实例 → 管理员）

| 项目 | 说明 |
|------|------|
| **位置** | `src/gateway/mod.rs`、`src/gateway/api.rs` |
| **端点** | `POST /api/report`（或 `/api/instances/:id/report`）：body 含 instance_id、类型（progress/alert）、payload；需认证（bearer 或实例级 token） |
| **存储** | 管理员侧可持久化到 `instances/admin/workspace/state/reports.jsonl` 或内存队列，供后续 CEO 轮询或推送 |

**代码修改要点**：

- Gateway 路由增加 `POST /api/report`，解析 body，校验来源，写入管理员 state 或转发给 CEO 逻辑（阶段 2/3 再接 CEO）。
- 若当前为单实例或非 admin gateway，可返回 404 或 403。

**单元测试**：

- `gateway::api::tests::report_endpoint_requires_auth`
- `gateway::api::tests::report_endpoint_stores_payload`

### 4.4 管理员 → CEO 消息

| 项目 | 说明 |
|------|------|
| **位置** | Gateway API |
| **端点** | `POST /api/admin-message`：管理员向指定实例的 CEO 发送 instruction/approval/rejection/info；body 含 `instance_id`、`message_type`、`payload` |
| **传递** | 阶段 2 实现 CEO 后，该消息写入目标实例的「CEO 收件箱」或 MessageBus 的 admin 通道；阶段 1 可仅落库或入队，不做消费 |

**代码修改要点**：

- 新增 `POST /api/admin-message`，校验调用方为管理员，将消息写入目标实例的存储（如 `instances/<id>/workspace/state/admin_messages.jsonl`）。
- 阶段 2 CEO 循环中读取该队列并注入到 CEO 上下文。

**单元测试**：

- `gateway::api::tests::admin_message_rejects_non_admin`
- `gateway::api::tests::admin_message_persists_for_instance`

### 4.5 阶段 1 交付清单与验收

| 交付物 | 验证方式 |
|--------|----------|
| 实例 CRUD + 端口分配 | 单元测试 instance_manager |
| `multiclaw instance` 子命令 | CLI/集成测试 |
| POST /api/report | 单元测试 gateway api |
| POST /api/admin-message | 单元测试 + 持久化校验 |
| 全量 | `cargo test` 通过 |

## 七、风险与测试策略

| 风险 | 缓解 |
|------|------|
| 配置解析与实例路径错误 | 大量 `config::schema` 与 `resolve_*` 单元测试；集成测试用 temp dir 模拟集群根 |
| 注册表与磁盘不同步 | 写后读、并发用单线程或锁；测试覆盖 save/load 与 add/update |
| 端口冲突 | 端口池从注册表推导；测试中使用高端口或 mock |
| 单实例回归 | 所有新分支在「无 instances.json」下保持现有行为；CI 保留现有单实例用例 |

每完成一个**子模块**（如 instance_registry、instance_manager、report 端点），即运行 `cargo test`，确保该模块测试通过且无回归。
