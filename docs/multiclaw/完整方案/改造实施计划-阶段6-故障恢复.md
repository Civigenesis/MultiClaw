# 改造实施计划 — 阶段 6：业务级故障恢复

> 依赖：阶段 3、5。实现业务级检查点、状态恢复、重启无缝衔接。每模块需单元测试。

---

## 一、检查点内容（与架构 10.1 对齐）

| 状态 | 说明 | 存储形式 |
|------|------|----------|
| 任务队列 | 待分配、进行中、阻塞 | 序列化 TaskQueue 或等价结构 |
| 任务执行进度 | 当前 turn、工具调用中、部分结果 | 按 task_id + entity_id 的进度记录 |
| 收件箱 | 未读消息 | MessageBus 持久化已有；检查点引用或快照 |
| 对话缓冲区 | 进行中的 1:1/群聊 | ConversationBus 持久化 + 检查点引用 |
| 记忆上下文 | 当前加载摘要 | 可选：仅记录「最近 N 条 key」引用 |
| 实体运行态 | 当前 prompt、history 快照 | 按 entity_id 的 snapshot |
| CEO 循环状态 | 上次分析时间、待处理决策 | 单条 CEO 状态结构 |

### 1.1 数据结构

| 项目 | 说明 |
|------|------|
| **新模块** | `src/checkpoint/mod.rs`，`src/checkpoint/snapshot.rs` |
| **CheckpointSnapshot** | 含上述各块的序列化；version 字段用于兼容；timestamp |
| **全量 vs 增量** | 全量：完整快照；增量：自上次全量以来的变更（如仅变更的 task、inbox 条、实体 state） |

## 二、检查点策略（架构 10.2）

| 策略 | 触发 | 实现要点 |
|------|------|----------|
| **全量** | 周期（如 5 分钟） | 后台任务：收集 MessageBus、ConversationBus、EntityPool 当前状态、Cron 下次触发、CEO 状态 → 序列化到 `instances/<id>/checkpoints/full_<ts>.json` |
| **增量** | 自上次全量后的变更 | 在关键路径（task 完成、inbox 变更、实体 turn 结束）写 delta 到 `checkpoints/delta_<ts>.json`；或合并到单一日志 |
| **关键点** | 任务完成、对话轮次结束 | 同增量，或触发一次「小检查点」 |

**代码修改**：

- `src/daemon/checkpoint_writer.rs`：周期全量 + 接收增量事件；写文件；保留最近 K 个全量 + 其间增量（或按时间窗口清理）。
- **单元测试**：`checkpoint::tests::full_snapshot_roundtrip`，`checkpoint::tests::delta_applied_after_full`

## 三、恢复流程（架构 10.3）

1. **启动时检测**：若存在 `instances/<id>/checkpoints/` 下最近全量文件，则进入恢复模式。
2. **加载**：读最近全量 + 其后的增量（按时间顺序 apply）。
3. **恢复**：EntityPool、MessageBus、ConversationBus、任务队列、收件箱 从快照重建（或从持久化 + 快照修正）。
4. **实体/CEO**：恢复当前 turn 的 prompt/history；若上次在工具执行中，可选重试或跳过并记录。
5. **继续运行**：Cron、Heartbeat、ProactiveScheduler 从恢复后的状态继续；记录一条 `health_update` 或 `recovery_completed` 事件。

**代码修改**：

- `src/daemon/mod.rs`：启动时若 instance 且存在 checkpoint，先调用 `checkpoint::load_latest_and_apply()`，得到恢复后的 MessageBus/ConversationBus/EntityPool 等，再 spawn gateway、scheduler 等。
- **单元测试**：`checkpoint::tests::recover_restores_inbox_and_tasks`，`daemon::tests::start_with_checkpoint_resumes_scheduler`

## 四、恢复策略（架构 10.4）

| 操作 | 说明 |
|------|------|
| **instance_restart** | CLI 或 API：发信号给 daemon 或子进程，触发优雅退出（先写一次检查点）再由 systemd/进程管理器拉起；或 kill + 外部重启 |
| **回滚** | 加载指定时间点的全量检查点，丢弃其后的增量；需谨慎（可能丢失后续进度） |
| **instance_create 重建** | 删除实例目录后重新 create；无检查点恢复，全新实例 |

**单元测试**：

- `checkpoint::tests::rollback_to_earlier_full_discards_later_deltas`
- 集成：`instance restart` 后 daemon 能读到最新检查点并恢复（可选）

## 五、与 Memory 一致性

- 检查点中**不**持久化完整 memory 内容，仅记录「当前会话引用的 memory key 或摘要 id」；恢复时用现有 Memory 后端按 key 再取。
- 若 Memory 支持按 scope 查询，恢复后首次 recall 按 entity/team/instance 拉取即可。
- **单元测试**：`checkpoint::tests::snapshot_references_memory_keys_only`，恢复后 `memory_recall` 仍返回预期 scope 数据（mock memory）。
