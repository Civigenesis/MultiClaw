## 一、事件与存储

### 1.1 事件类型（与架构 8.4 对齐）

| 事件 | 维度 | 说明 |
|------|------|------|
| task_created | instance_id, entity_id?, team_id?, timestamp | 任务创建 |
| task_assigned | instance_id, entity_id, task_id, timestamp | 任务分配 |
| task_completed | instance_id, entity_id, task_id, timestamp | 任务完成 |
| turn_start / turn_end | instance_id, entity_id, timestamp | Agent 轮次 |
| tool_call | instance_id, entity_id, tool_name, timestamp | 工具调用 |
| report | instance_id, type, timestamp | 汇报 |
| approval_request | instance_id, type (skill/global_share/resource), timestamp | 审批请求 |
| health_update | instance_id?, component, status, timestamp | 健康状态 |

### 1.2 采集与聚合

| 项目 | 说明 |
|------|------|
| **新模块** | `src/observability/events.rs` 或 `src/observability/cluster_events.rs` |
| **接口** | `emit(event: ClusterEvent)`；内部写入本地 buffer 或直接写存储；支持从 gateway、daemon、agent、cron 等处调用 |
| **存储** | 原始事件：7 天（可配置），如 `instances/admin/workspace/state/events/YYYY-MM-DD.jsonl`；聚合：30 天，如按 instance/entity 的日汇总表或同目录下 aggregate_*.json |
| **配置** | `[observability]` 或新 `[observability.events]`：retention_days_raw=7, retention_days_agg=30, enabled=true |

**单元测试**：

- `observability::events::tests::emit_persists_to_daily_file`
- `observability::events::tests::aggregation_by_instance_and_day`
- `observability::events::tests::retention_cleans_old_files`

### 1.3 与现有 Observer 集成

| 位置 | 说明 |
|------|------|
| `src/observability/traits.rs` | `Observer` 已有 `record_event`；可扩展事件枚举，或新增 `Observer::record_cluster_event(ClusterEvent)`；默认实现将 ClusterEvent 转为现有 ObserverEvent 并写入 events 模块 |
| **单元测试** | `observability::tests::observer_forwards_cluster_events` |

## 二、REST API（可观测）

| 端点 | 用途 | 数据 |
|------|------|------|
| GET /api/observability/events | 查询事件 | query: instance_id, entity_id, from, to, type；分页 |
| GET /api/observability/summary | 汇总 | 按实例/实体/日的计数、健康 |
| GET /api/instances | 实例列表 | 来自 InstanceRegistry + 健康状态 |
| GET /api/instances/:id | 实例详情 | 配置摘要、实体数、最近事件 |
| GET /api/approvals | 待审批列表 | skill_approval_pending + global_share_pending + resource_requests |

**代码修改**：

- `src/gateway/api.rs`：注册上述路由；读取 events 模块与 InstanceRegistry、pending 文件；返回 JSON。
- **单元测试**：`gateway::api::tests::observability_events_endpoint_returns_filtered`，`approvals_endpoint_returns_pending_list`

## 三、Web 仪表盘

### 3.1 路由（与架构 8.3 对齐）

| 路由 | 内容 |
|------|------|
| /login | 认证（复用现有 pairing 或扩展） |
| /dashboard | 用户/管理员：资源、实例列表、待审批、汇报时间线 |
| /instances/:id | 实例详情 |
| /instances/:id/ceo | CEO 视图：团队、任务、实体 |
| /instances/:id/teams/:tid | 团队看板 |
| /instances/:id/entities/:eid | 实体执行台 |
| /approvals | 审批中心（技能、全局共享、资源） |

### 3.2 实现方式

| 项目 | 说明 |
|------|------|
| **前端** | 复用现有 `web/`（React/TS）；新增路由与页面组件；调用上述 REST API |
| **图表** | 甘特图、看板、燃尽图：引入 ECharts 或专用 Gantt/Board 库；数据来自 /api/observability/events 与 /api/instances/:id 等 |
| **静态与 Gateway** | 现有 gateway 已可挂静态资源；确保 SPA 路由 fallback 到 index.html |

**单元测试**：

- 前端：组件与 API 调用的 Jest/Vitest 单元测试（若项目已有）；或 E2E 可选。
- 后端：上述 API 的单元测试已覆盖。

## 四、记忆作用域实现（架构 9.1）

| 项目 | 说明 |
|------|------|
| **Key 前缀** | entity:<id>:, team:<id>:, instance:；在 memory store/recall 时由调用方传入 scope，或从当前 entity/team 推导 |
| **位置** | `src/memory/sqlite.rs`（或当前 backend）：store 时 key 带前缀；recall 时按 scope 过滤 |
| **访问控制** | 读/写时校验：实体只能写自身与所属 team/instance；CEO 可读全实例 |
| **单元测试** | `memory::sqlite::tests::store_with_entity_scope_prefix`，`recall_filters_by_scope`，`reject_write_wrong_scope` |
