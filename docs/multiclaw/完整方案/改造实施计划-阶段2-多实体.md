## 一、Config 扩展 `[instance]`

### 1.1 数据结构（schema）

| 位置 | 修改内容 |
|------|----------|
| `src/config/schema.rs` | 新增 `InstanceConfig` 结构体（或内嵌到 `Config` 的 `instance: Option<InstanceConfig>`） |

**字段**（与架构 4.1 对齐）：

```rust
// 新增或扩展现有
pub struct InstanceConfig {
    pub preset: Option<String>,           // startup | enterprise | brainstorm | freeform | project
    pub default_provider: Option<String>,
    pub default_model: Option<String>,
    pub ceo: Option<CeoConfig>,          // entity_id=ceo 的配置
    pub entities: Vec<EntityConfig>,
    pub teams: Vec<TeamConfig>,          // enterprise / project
    pub projects: Vec<ProjectConfig>,     // project preset
}

pub struct CeoConfig {
    pub enabled: Option<bool>,
    // 可扩展：provider/model 覆盖
}

pub struct EntityConfig {
    pub id: String,
    pub provider: Option<String>,
    pub model: Option<String>,
    pub team_id: Option<String>,
    pub role: Option<String>,
    pub skills: Option<Vec<String>>,
}

pub struct TeamConfig {
    pub id: String,
    pub name: Option<String>,
}

pub struct ProjectConfig {
    pub id: String,
    pub name: Option<String>,
}
```

- 在 `Config` 中增加 `pub instance: Option<InstanceConfig>`，`#[serde(default)]`。
- 若 `instance` 为 None 或 preset 未设置，视为单实体（当前行为）或默认 `startup`。

**单元测试**：

- `config::schema::tests::instance_config_deserializes_preset_startup`
- `config::schema::tests::instance_config_entities_with_optional_provider_override`
- `config::schema::tests::instance_config_absent_defaults_to_none`

### 2.3 CEO 工具集

| 项目 | 说明 |
|------|------|
| **位置** | `src/tools/` 下新增或扩展现有 |
| **工具** | `create_team`, `create_entity`, `assign_task`, `instance_status` 等；**不**提供 `instances_list` 等实例管理能力（仅管理员） |
| **权限** | 仅当当前运行身份为 `entity_id=ceo` 时，这些工具在 tool list 中暴露并可用 |

**代码修改要点**：

- 新增 `src/tools/ceo_create_team.rs`、`ceo_create_entity.rs`、`ceo_assign_task.rs`、`ceo_instance_status.rs`（或统一 `ceo_tools.rs`）。
- 每个工具：校验 `current_entity_id == "ceo"`；**create_team / create_entity 必须持久化到当前实例的 config.toml**（追加 `[instance.teams]` / `[[instance.entities]]` 并调用 `Config::save()`），不得仅为占位或仅写内存/state。

- 在 `src/tools/mod.rs` 中根据「当前实体」条件注册上述工具；agent 调用时传入 `current_entity_id`；CEO 工具需持有 `config_path`、`workspace_dir` 以执行加载/保存与目录创建。

**单元测试**：

### 2.4 单实例多实体：实体/团队 workspace 管理（阶段 2 必做）

每个成员（实体）及团队应有独立的工作空间，便于隔离记忆、状态、技能与产出。

| 项目 | 说明 |
|------|------|
| **目录约定** | 实例 workspace 根为 `cluster_root/instances/<instance_id>/workspace/`。其下：`entities/<entity_id>/` 为实体专属目录；`teams/<team_id>/` 为团队共享目录（可选）。CEO 使用实例级 workspace 或 `entities/ceo/` 均可，由实现统一约定。 |
| **实体 workspace** | `workspace/entities/<entity_id>/` 下可包含：`memory/`、`state/`、`sessions/`、`skills/` 等子目录（与实例级结构一致或子集）。Agent 以某实体身份运行时，记忆、会话、状态应使用该实体目录；未指定实体时使用实例级 `workspace/`。 |
| **团队 workspace** | `workspace/teams/<team_id>/` 用于该团队共享产出或共享记忆前缀 `team:<id>:`；create_team 落盘后创建此目录。 |
| **解析与注入** | 在 `Config` 或 agent 入口根据 `target_entity_id` 解析 `entity_workspace_dir`（即 `workspace_dir.join("entities").join(entity_id)`）；Memory/State 等使用该路径作为根，或在使用 instance 级路径时加 scope 前缀 `entity:<id>:`。 |
| **create_entity 时** | 创建新实体时除写 config 外，必须 `create_dir_all(workspace_dir/entities/<entity_id>)`，并可初始化子目录 `memory`、`state` 等。 |

**单元测试**：

- `config::schema::tests::entity_workspace_path_resolution`
- `tools::ceo::tests::create_team_persists_to_config`
- `tools::ceo::tests::create_entity_persists_and_creates_workspace`

### 2.5 每实体的个性化配置与描述文件（独立决策能力）

为实现单实例多实体中**每个实体独立决策**，每个实体（含 admin 实例、CEO、各工作节点）除在 config 中注册名称外，应拥有自己的**描述文件**与可选的**个性化配置**，用于构建该实体的 system prompt（身份、行为、边界）。

| 项目 | 说明 |
|------|------|
| **当前实现** | 实例级单一 `workspace_dir` 下存在 OpenClaw 风格文件：`AGENTS.md`、`SOUL.md`、`TOOLS.md`、`IDENTITY.md`、`USER.md`、`HEARTBEAT.md`、`BOOTSTRAP.md`、`MEMORY.md`；以及 config 中 `[identity]`（如 AIEOS `aieos_path`/`aieos_inline`）。这些在 `build_system_prompt` 时从 `workspace_dir` 注入，形成「你是谁、如何行为」的上下文。 |
| **目标** | 每个实体有**自己的** prompt 根目录（即其实体 workspace）：该目录下可放置上述描述文件（至少 `IDENTITY.md`、`SOUL.md`、`AGENTS.md`；可选 `agent.md` 作为别名）。当以该实体身份运行时，system prompt 从**该实体的 workspace** 加载这些文件；缺失时可按需回退到实例级 workspace。 |
| **与 config 的关系** | 实例仍共用一份 `config.toml`（含 `[instance]`、`[[instance.entities]]`）；每实体的「身份/人格」由其实体 workspace 内的描述文件表达，不要求每实体单独一份 config 文件。若需每实体独立 AIEOS，可在实体 workspace 内放置 `identity.json`，并在后续扩展中支持 per-entity `identity_path`。 |
| **初始化** | **实例创建时**：可为实例级 workspace 做一次最小 scaffold（`IDENTITY.md`、`SOUL.md` 等），供未指定 entity 或 CEO 使用实例级目录时使用。**实体创建时**（create_entity）：在 `workspace/entities/<entity_id>/` 下执行 **scaffold_entity_workspace**，写入默认 `IDENTITY.md`、`SOUL.md`、`AGENTS.md`（内容以 entity_id / role 为名），使新实体从创建起即具备独立人格描述。 |
| **Prompt 解析** | 当 `target_entity_id` 为 Some 时，**effective_prompt_workspace** = `workspace_dir.join("entities").join(entity_id)`；将 `effective_prompt_workspace` 传入 `build_system_prompt_with_mode`（及 identity 加载），实现「按实体加载描述文件」。未指定实体时仍使用实例级 `workspace_dir`。 |

**单元测试**：

- `entity::tests::scaffold_entity_workspace_creates_identity_soul_agents`
- `agent::loop_::tests::prompt_uses_entity_workspace_when_target_entity_set`

## 四、五类 preset 行为（配置驱动）

| preset | 配置表现 | 代码侧 |
|--------|----------|--------|
| startup | 无 teams；可选 1:1/群聊 | Config 解析；EntityPool 不要求 team_id |
| enterprise | teams、role、team_id | EntityConfig 含 team_id；assign_task 可指定 team |
| brainstorm | 无层级；conversation 权重高 | 主要影响阶段 3 的 ConversationBus 与调度权重 |
| freeform | random_social、allow_organic_teams | 阶段 3 使用 |
| project | projects、project_ids | 任务绑定 project_id；阶段 3/5 看板用 |

**阶段 2 必做**：Config 能解析五类 preset 对应字段；EntityPool/Entity 能按 `team_id`、`role` 区分。其余（conversation 权重、random_social）在阶段 3 实现。

**单元测试**：

- `config::schema::tests::preset_enterprise_parses_teams_and_roles`
- `config::schema::tests::preset_project_parses_projects`
